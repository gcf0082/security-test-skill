# 安全测试流程：基于正常业务流量的攻击模拟分析

## 1. 前置准备
- **请求 ID 优先**：每次请求（业务或攻击）后从响应头读取 `X-Request-Id`，直接作为 `http_req_query` 工具的查询参数即可拿到该请求的完整上下文（请求/响应、后端处理日志、服务调用链、性能指标等），这是首选追踪路径。
- **时间记录**：同时记录每次操作的开始与结束时间。当响应未返回 `X-Request-Id` 时，用时间范围调 `http_req_query` 作为后备查询；也用于判定其他事件是否与本次操作相关。

## 2. 正常业务功能测试
- 按标准用户流程完成一次完整业务操作（登录/下单/提交表单等），仅使用正常业务输入，不含任何攻击载荷。
- 为每次请求记录其 `X-Request-Id`，与操作步骤一一对应。

## 3. 获取并追踪请求信息
- **首选**：用 `X-Request-Id` 调 `http_req_query`，一次命中拿到完整细节。
- **后备**：用时间范围调 `http_req_query` 查业务操作对应的请求记录，取其请求 ID 后再通过 `trace_request`（或再次调 `http_req_query`）获取详情。

## 4. 攻击模式识别
- 读 `攻击模式特征/Web页面特征.md`（页面层面）与 `攻击模式特征/HTTP请求特征.md`（请求/响应层面），匹配当前业务场景对应的攻击模式。
- 匹配后加载 `攻击模式库/黑灰盒攻击模式/<攻击名>/测试方法.md` 获取执行策略。

## 5. 构造并发送攻击请求
- **Header 保持**：除特殊情况外不得修改原始请求的 HTTP 头部字段。
- **Payload 修改**：仅对请求体（Body）做最小必要改动以注入攻击载荷。
- **发送工具**：

  | 场景 | 工具 |
  |------|------|
  | 发送独立 HTTP 请求 | `python requests` / `curl` |
  | 复杂浏览器交互（点击、重定向链、执行 JS 等） | Chrome DevTools MCP 或浏览器自动化 |

  仅发送已构造好的攻击请求时，不要使用浏览器自动化工具，以免引入不必要的环境开销。

## 6. 输出与记录
- 记录攻击请求完整内容（URL、方法、Headers、Body）、响应状态码与关键响应内容、响应头中的 `X-Request-Id`。
- 结合 `http_req_query`（以 `X-Request-Id` 为参数）获取的追踪信息，进行攻击效果分析与问题定位。

## 7. 判定证据要求

> 凡攻击模式的"成立"定义涉及后端可观测状态变化（文件写盘、解压落点、DB 写入、命令执行、出网回连、配置激活、越权读/写等），必须按 `核心策略/置信度与升级策略.md` 的判定标准与升级流程执行。

详细规则——证据来源（后端独立通道 / 响应自证）、置信度等级、判定矩阵、升级清单、按攻击类型推荐路径、终止条件——全部见该文件。本文件不重复定义。
