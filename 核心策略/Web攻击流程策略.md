# 安全测试流程：基于正常业务流量的攻击模拟分析

## 1. 前置准备
- **请求 ID 优先**
  每次发送请求（业务请求或攻击请求）后，**必须**尝试从响应头读取 `X-Request-Id`（大小写不敏感，常见变体：`X-Request-ID`、`X-Trace-Id`、`Request-Id`）。
  拿到的 `X-Request-Id` 是后续查询该请求详细信息的**首选凭据**——直接把它作为参数传给 `myproxy-query` skill，即可精确拿到这一次请求的完整上下文（完整请求/响应、后端处理日志、服务调用链、性能指标等）。
  **这是最可靠的追踪路径**，优先于时间范围查询。
- **时间记录要求（后备方案）**
  每次执行正常业务功能测试、攻击开始结束和关键操作节点，**必须**记录操作的开始时间与结束时间。
  用途：
  1. 当响应未返回 `X-Request-Id` 时，用时间范围向 `myproxy-query` 查询该业务操作产生的请求记录。
  2. 判定关键事件是否与本次操作相关——不在时间区间内的事件大概率与本次无关。

## 2. 正常业务功能测试
- 按照标准用户流程完成一次完整的业务功能操作（例如登录、下单、提交表单等）。
- 操作过程中**仅使用正常业务输入**，不包含任何攻击载荷。
- **每次请求的响应头 `X-Request-Id` 必须记录下来**，与操作步骤一一对应。

## 3. 获取并追踪请求信息
**首选路径（有 X-Request-Id 时）**：
- 直接调用 `myproxy-query` skill，以 `X-Request-Id` 作为查询参数。
- 获取该请求的完整细节（请求体、响应体、后端处理链路、下游调用、耗时等）。
- 无需时间范围匹配，一次命中。

**后备路径（无 X-Request-Id 时）**：
- 调用 `myproxy-query` skill，基于第 1 步记录的时间范围查询该业务操作所产生的 HTTP 请求。
- 从结果中提取目标请求的**唯一请求 ID**。
- 使用该请求 ID 传给 `trace_request` 工具（或再次调用 `myproxy-query`）获取完整追踪信息。

无论走哪条路径，最终目的都是拿到"这一次请求"的完整上下文数据，用于支撑后续的深入分析。

## 4. 攻击模式识别
- 从项目根目录下的以下文件中读取并匹配攻击特征：
  - `攻击模式特征/Web页面特征.md`   → 用于识别页面层面的攻击模式
  - `攻击模式特征/HTTP请求特征.md` → 用于识别请求/响应层面的攻击模式
- 根据特征描述确定当前业务场景对应的潜在攻击模式及改造策略。
- 匹配到攻击模式后，加载 `攻击模式库/黑灰盒攻击模式/<攻击名>/执行方法.md` 获取具体执行策略。

## 5. 构造并发送攻击请求
- **构造规则**：
  - **Header 保持原则**：除特殊情况外，**不得修改**原始请求的 HTTP 头部字段。
  - **Payload 修改原则**：仅对请求体（Body）进行最小必要改动以注入攻击载荷。

- **发送工具选择原则**：
  | 场景类型 | 推荐工具 | 说明 |
  |----------|----------|------|
  | 发送简单的独立 HTTP 请求 | `python requests` 库 或 `curl` 命令行工具 | 仅需发送一次请求并获取响应，无浏览器上下文依赖 |
  | 涉及复杂浏览器交互的测试场景 | Chrome DevTools MCP 或其他浏览器自动化工具 | 例如需模拟用户点击、处理重定向链、执行 JavaScript 代码等复杂流程 |

  **注意**：若测试目标仅为发送一个已构造好的攻击请求，不应使用浏览器自动化工具，以避免引入不必要的环境变量和性能开销。

- **攻击请求也要捕获 `X-Request-Id`**：
  攻击请求发送后同样从响应头读取 `X-Request-Id` 并记录；它是定位该次攻击对应后端处理细节（是否触发 WAF、是否进入业务逻辑、是否抛异常等）的首选凭据。

## 6. 输出与记录
- 记录发送的攻击请求完整内容（URL、方法、Headers、Body）。
- 记录服务端响应状态码、关键响应内容、以及**响应头中的 `X-Request-Id`**（若存在）。
- 结合从 `myproxy-query`（以 `X-Request-Id` 为参数）或 `trace_request` 获取的完整追踪信息，进行攻击效果分析与问题定位。
- 报告中对每一次关键请求都应附上其 `X-Request-Id`，方便后续复查。
