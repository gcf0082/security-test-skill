# 安全测试流程：基于正常业务流量的攻击模拟分析

## 1. 前置准备
- **请求 ID 优先**：每次请求（业务或攻击）后从响应头读取 `X-Request-Id`，直接作为 `myproxy-query` skill 的查询参数即可拿到该请求的完整上下文（请求/响应、后端处理日志、服务调用链、性能指标等），这是首选追踪路径。
- **时间记录**：同时记录每次操作的开始与结束时间。当响应未返回 `X-Request-Id` 时，用时间范围调 `myproxy-query` 作为后备查询；也用于判定其他事件是否与本次操作相关。

## 2. 正常业务功能测试
- 按标准用户流程完成一次完整业务操作（登录/下单/提交表单等），仅使用正常业务输入，不含任何攻击载荷。
- 为每次请求记录其 `X-Request-Id`，与操作步骤一一对应。

## 3. 获取并追踪请求信息
- **首选**：用 `X-Request-Id` 调 `myproxy-query`，一次命中拿到完整细节。
- **后备**：用时间范围调 `myproxy-query` 查业务操作对应的请求记录，取其请求 ID 后再通过 `trace_request`（或再次调 `myproxy-query`）获取详情。

## 4. 攻击模式识别
- 读 `攻击模式特征/Web页面特征.md`（页面层面）与 `攻击模式特征/HTTP请求特征.md`（请求/响应层面），匹配当前业务场景对应的攻击模式。
- 匹配后加载 `攻击模式库/黑灰盒攻击模式/<攻击名>/测试方法.md` 获取执行策略。

## 5. 构造并发送攻击请求
- **Header 保持**：除特殊情况外不得修改原始请求的 HTTP 头部字段。
- **Payload 修改**：仅对请求体（Body）做最小必要改动以注入攻击载荷。
- **发送工具**：

  | 场景 | 工具 |
  |------|------|
  | 发送独立 HTTP 请求 | `python requests` / `curl` |
  | 复杂浏览器交互（点击、重定向链、执行 JS 等） | Chrome DevTools MCP 或浏览器自动化 |

  仅发送已构造好的攻击请求时，不要使用浏览器自动化工具，以免引入不必要的环境开销。

## 6. 输出与记录
- 记录攻击请求完整内容（URL、方法、Headers、Body）、响应状态码与关键响应内容、响应头中的 `X-Request-Id`。
- 结合 `myproxy-query`（以 `X-Request-Id` 为参数）获取的追踪信息，进行攻击效果分析与问题定位。

## 7. 判定证据要求（后端状态核验 + 置信度定级）

> 本节是**黑灰盒**与**黑白盒结合**两种模式的通用门槛；白盒模式按 SKILL.md Step B 跳过本文件，不受影响。

### 7.1 适用范围

凡该攻击模式的"成立"定义涉及**后端可观测状态变化**的，必须执行后端核验后才能定论。常见类别：
- 文件系统：上传落点、解压落点、临时文件写入
- 数据存储：DB 增删改、缓存写入、配置激活
- 进程/网络：命令执行、出网回连、内部服务调用
- 业务状态：越权读/写订单、账号、权限等

### 7.2 置信度等级

置信度与漏洞级别**并列**，衡量「我们对该漏洞真实可利用的把握程度」。

**核心边界**：HTTP 响应（无论响应码、响应体多么"像成功"）属于请求层面观察；`myproxy-query` 查到的也是 HTTP 请求/响应记录。两者均**不构成后端状态证据，不用于置信度评估**。

| 置信度 | 定义 | 典型证据 |
|--------|------|----------|
| **已确认** | 后端状态被独立可观测通道直接证实，可脱离 HTTP 响应独立复现 | 业务接口返回了越界产物内容；`myssh` 在服务器上直接观测到越界文件/进程；带外通道收到回连（DNSlog、HTTP callback） |
| **中** | 响应侧行为强烈指向漏洞，但后端实际状态未经独立核验 | 响应体本身包含只有漏洞触发才能取得的数据（越权数据、越界路径内容泄漏） |
| **低** | 理论路径存在，无运行时证据 | 仅基于代码审计/配置推断；黑灰盒尝试均被拒绝但不能确认是否完全过滤 |

**强约束**：「成立」结论的置信度**必须是「已确认」**。置信度「中」或「低」时只能判「不确定」，进白盒补充流程，不得出「成立」报告。

### 7.3 后端核验通道

| 通道 | 可达置信度 | 适用条件 |
|------|-----------|----------|
| 业务侧只读接口（列表/详情/状态查询）返回了可验证的越界产物内容 | **已确认** | 目标系统暴露了相关查询接口 |
| `myssh` 只读命令（`ls`/`stat`/`find`）直接观测越界文件/进程 | **已确认** | `环境配置/测试环境.md` 明确允许 SSH 访问 |
| 带外通道收到回连（DNSlog、HTTP callback） | **已确认** | 盲 SSRF/RCE/XXE 等无直接回显的漏洞 |
| 响应体直接包含越界产物内容（如越权数据原文） | **中** | 无独立核验通道时的次优选，仍只能判「不确定」 |

### 7.4 判定矩阵

| 后端状态证据 | 结论 | 置信度 |
|---|---|---|
| 独立通道直接证实越界影响存在 | **成立** | 已确认 |
| 独立通道证实后端无越界（payload 被过滤/路径被规整） | **不成立** | 已确认 |
| 仅响应侧异常，后端状态未经独立通道核验 | **不确定** | — |
| 无任何证据 | **不确定** | — |

「不确定」项不填置信度；按主策略走白盒补充再回灌，直到结论收敛至「成立/不成立」。
