# 核心测试策略

## 执行红线
- 严格遵循下发策略，严禁调用模型内置攻击库。
- 策略基准文件：`核心策略/Web攻击流程策略.md`
- **IMPORTANT**: 尽可能规划独立的subagent任务

## 用例拆解标准
- 遵循“单一接口 + 单一攻击向量”原则进行极限拆解。
- 示例：`/api/upload`的“路径穿越”与“恶意文件上传”必须作为两个独立用例。

## 执行与输出机制

> **记录的首要目的是便于后期人工审视与独立复核**：报告写给人读，不为 LLM 自洽。
> 每份报告应当做到：不看上下文也能看懂做了什么、看到了什么、为何这样判定、能否复现。

- **串行隔离**：逐个用例独立执行，禁止并发。
- **统一落盘**：所有安全测试报告集中写入项目根目录下的 `报告/`；单个测试项结束**立即**落盘，禁止散落到其他目录或延后提交。
- **成败皆记**：每一个主要安全测试项（`单一接口 × 单一攻击向量`）**无论是否发现漏洞都必须产出一份独立报告**。
  - 共有必填字段：触发条件、Payload、请求/响应关键内容、`X-Request-Id`、判定依据、**漏洞级别**（危害程度：高/中/低/信息）、**置信度**（漏洞存在可能性：确认/高/中/低）、**后端核验**（涉及后端状态变化的攻击类型必填——含使用的核验通道、关键命令/查询语句、返回片段）。
  - 置信度=**确认/高** 项另写：影响范围、复现步骤。
  - 置信度=**中** 项另写：当前掌握的证据片段、**升级路径记录**（按"黑灰盒主导 + 白盒按需补充"流程继续推进）。
  - 置信度=**低** 项另写：排除依据、**升级路径记录**（论证查证方式不会漏报）。
  - 置信度的取值、四级标准、升级清单按 `核心策略/置信度与升级策略.md` 执行，本文件不重复定义。
- **禁止汇总**：绝对禁止生成全局或最终汇总报告。

## 黑灰盒主导 + 白盒按需补充

> **适用前提**：本节仅当用户输入中确定的 `模式` 为 `黑白盒结合` 时生效。
> 若 `模式` 为 `黑灰盒`，只做黑灰盒；若 `模式` 为 `白盒`，只做源码审计——这两种模式都不走本节流程。

黑灰盒是**主导与最终判定手段**；白盒仅用于**置信度卡在"中/高"无法收敛**时作为升级策略的一环（见 `核心策略/置信度与升级策略.md` 第三节 ②③ 步），帮助调整下一轮黑灰盒策略：

1. **黑灰盒测试**：基于外部输入输出（抓包、响应、接口文档、日志等）进行探测，得到一个置信度等级。
   - **确认 / 低**：直接落盘——确认=漏洞认定；低=查证彻底未发现问题。
   - **高 / 中**：进入升级流程（按 `核心策略/置信度与升级策略.md` 第三节执行）。
2. **升级流程**含白盒读代码这一步——结合源码了解入口、数据流、过滤/校验、下游调用，找出可调整的下一轮黑灰盒方向。白盒自身**不下最终判定**。
3. **回灌黑灰盒**：根据升级所得，改造 Payload 或测试条件，再走第 1 步。直到收敛到"确认 / 低"，或穷尽合理路径仍停在"中/高"。

最终每一项的落盘结论**只能来自黑灰盒**（见 `执行与输出机制 · 成败皆记`）。若引入过白盒补充，应作为该报告的"附带分析"段落附上关键代码片段与得出的测试决策。
