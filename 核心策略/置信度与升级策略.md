# 漏洞置信度与升级策略

> 适用：黑灰盒、黑白盒结合（白盒按 SKILL.md Step B 跳过）。本文件是置信度规则的唯一权威。

## 一、什么算"真有漏洞"——证据来源

要判定漏洞，需要拿到证据。证据有两种来源：

- **A 去后端实地查看**：调业务的查询接口看后端数据；用 `myssh` 登服务器看文件（`ls`/`stat`/`find` 等只读命令）；让漏洞触发时往外发请求，DNSlog 或 HTTP callback 收到回连就证明发生过。
- **B 响应自己暴露了证据**：漏洞产物直接出现在响应里，能一眼认出。比如读 `/etc/passwd` 的 payload，响应吐出 `root:x:0:0:...` 这种文件本身内容；SQL 注入响应里出现具体表名/数据库版本/用户哈希；XXE/SSRF 直接回显内部文件或内网响应；反射型 XSS 看到 payload 没被转义；越权时拿到别人具体可核对的订单/账号信息。

> 注意：响应里只是"行为有差异"（布尔差/时间差/模糊错误）不算自证，必须能独立认出"这就是漏洞产物"。

## 二、置信度四级（漏洞存在的可能性）

每个测试项最终用一个等级表达，**不再分"结论 + 置信度"两个维度**。

| 等级 | 含义 | 典型证据 |
|------|------|----------|
| **确认** | 铁证，漏洞确实可被利用 | A 后端看到了越界产物；B 响应里有自证证据；带外回连成功 |
| **高** | 大概率存在但还缺铁证 | 响应有强指向但不够自证；多次尝试一致；代码审计强烈支持 |
| **中** | 有线索但不确定 | 布尔/时间差异、模糊错误信息、不可核对的小值；后端没看到产物但不能确定是真没发生还是没查到 |
| **低** | 大概率没问题 | 后端查证彻底没发现，且查证方式不会漏报；多次尝试均无任何异常 |

> 一个不对称的事实：**看到证据=一定发生了；没看到≠没发生**——可能没找对地方或没赶上时机。"低"必须解释清楚为什么这次查证不会漏报，否则降到"中"。
> **硬规矩**：**确认/高 = 漏洞有效**，必须出报告促修；**中 = 还需更多证据**；**低 = 暂未发现问题**。

## 三、查证不到时怎么办——升级清单

当前等级停在"中"或"高"（即既不能锤死也不能排除）时**禁止立即下结论**，按下表挨个试，每试一步重新评估等级。

| 步骤 | 做什么 |
|------|--------|
| ① 看 payload 真的到了关键代码 | 抓包/`myproxy-query` 确认 payload 没被前置过滤拦掉、确实走到了目标处理逻辑 |
| ② 翻代码看防御机制 | 找黑名单/正则/规整逻辑，看哪里能绕过 |
| ③ 翻代码看真实落点 | 数据实际写到哪个目录、哪张表，常常和我们以为的不是一个地方 |
| ④ 换更广的范围去查 | 按代码里看到的真实路径全扫一遍；多个 A 通道交叉验证 |
| ⑤ 抓瞬间存在的痕迹 | 文件刚创建就被删了？用 `inotifywait`/audit/高频轮询/前后 diff 抓现行 |
| ⑥ 加带外回连 | 让漏洞触发时往外发个 DNS 或 HTTP 请求，DNSlog 收到就是铁证 |
| ⑦ 用新发现回灌 | 把上面发现的东西做成新 payload，重做一次黑灰盒 |

**典型顺序**：路径穿越读 → B 自证→②→⑦；路径穿越写/Zip Slip → ④→③→⑤→⑥→⑦；盲注/SSRF/命令注入 → ⑥ 带外回连；XSS/越权 → B 自证→③。

**何时停下**：升到"确认"或降到"低"直接落盘；或跟本攻击相关的步骤都试遍了仍卡在"中/高"，按当前等级落盘——报告必须有"升级路径记录"段，写明每一步做了什么、结果如何。
