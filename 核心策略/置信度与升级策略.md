# 漏洞置信度与升级策略

> 适用范围：黑灰盒、黑白盒结合两种模式（白盒模式按 SKILL.md Step B 跳过本文件）。
> 本文件是置信度规则的唯一权威；其他文档只引用本文件，不重复定义。

## 一、判定证据的两类来源

成立结论必须基于"独立可验证的证据"。证据来源分两类：

### A. 后端独立通道

| 通道 | 适用条件 |
|------|----------|
| 业务侧只读接口（列表/详情/状态查询） | 目标系统暴露了相关查询接口 |
| `myssh` 只读命令（`ls`/`stat`/`find`/`auditctl` 等） | `环境配置/测试环境.md` 明确允许 SSH 访问 |
| 带外通道（DNSlog、HTTP callback） | 盲漏洞场景（盲 SSRF/RCE/XXE） |

### B. 响应自证

响应内容本身是漏洞利用的产物，且具独立可验证特征：

- **路径穿越读取**：响应是 `/etc/passwd`、`/etc/shadow` 等具有标志性结构的文件
- **SQL 注入回显**：响应是数据库元数据（表名、版本、用户哈希等）
- **XXE 回显**：响应是被读取的内部文件
- **SSRF 直回显**：响应是内网服务的具体响应
- **反射型 XSS**：payload 在响应中未转义
- **越权读**：返回了可独立核对的他人具体业务数据（订单详情、账号信息等）

> **关键限定**：内容必须可独立识别为漏洞产物。布尔差异、时间差异、模糊错误信息、不可核对的小值，**不算自证**。

## 二、置信度等级

| 置信度 | 定义 | 成立侧典型 | 不成立侧典型 |
|--------|------|-----------|-----------|
| **已确认** | 漏洞产生了独立可验证的证据 | A 类后端通道发现产物；或 B 类响应自证 | 核验方法在原理上覆盖所有可观测痕迹（如带外通道无回连） |
| **中** | 有指向性偏差但不具自证性，未做后端核验；或后端核验未发现但可能漏检 | 布尔/时间差异、模糊错误信息、不可核对的小值 | 核验已做但未发现，且不能排除漏检 |
| **低** | 仅理论路径，无运行时证据 | 仅基于代码审计/配置推断 | 黑灰盒尝试均被拒绝但未做后端核验 |

> 举证不对称：「找到产物」或「响应自证」一锤定音；「没找到」不等于不存在。「不成立 + 已确认」需要论证核验方法的覆盖完整性，否则最高给「中」。

**强约束**：「成立」必须是「已确认」（达不到则进升级流程，仍不行则判「不确定」）；「不成立」默认是「中」，升到「已确认」需显式论证覆盖完整性。

## 三、判定矩阵

| 证据状态 | 结论 | 置信度 |
|---|---|---|
| A 类后端通道发现越界产物 | **成立** | 已确认 |
| B 类响应自证 | **成立** | 已确认 |
| 后端核验未发现产物，核验方法能覆盖所有可观测痕迹（如带外通道无回连） | **不成立** | 已确认 |
| 后端核验未发现产物，但不能排除漏检 | **不成立** | 中 |
| 响应有偏差但不自证，未做后端核验 | **不确定** | — |
| 无任何证据 | **不确定** | — |

## 四、升级策略（核心 — 当置信度不足以下结论时必走）

**触发条件**：当目前最高置信度无法支持「成立(已确认)」或「不成立(已确认)」结论时——**禁止立即下结论**，按以下顺序逐项尝试。每尝试一项后回到第三节判定矩阵重新评估证据状态。

### 4.1 通用升级清单（按优先级）

| 步骤 | 做什么 | 适用场景 |
|------|--------|----------|
| ① 验证 payload 命中 | 抓包/`myproxy-query` 看 payload 是否被中间件改写、是否走到目标处理代码 | 任何置信度不足场景的第一步 |
| ② 白盒读过滤代码 | 定位防御机制（黑名单/白名单/正则/规整逻辑），理解"被拒"是哪段代码做的；据此判断是否有绕过空间 | 黑灰盒尝试均被拒 |
| ③ 白盒读处理代码找真实落点 | 反查实际写入路径、表名、调用链；与黑灰盒搜索范围对比，确认核验位置正确 | 后端核验未发现产物 |
| ④ 扩展核验范围 | 按白盒发现的真实路径全扫（不止预设目录）；跨多个 A 类通道交叉验证（业务接口/`myssh` 互补） | 核验范围可能不足 |
| ⑤ 监控瞬态痕迹 | 若代码显示文件/记录创建后立即清理：`inotifywait`/audit/`tail -f` 实时监控；攻击前后做目录 diff；高频轮询 | 代码或日志显示存在异步清理 |
| ⑥ 引入带外触发 | 没有响应自证、A 类核验也漏检时，加入 DNSlog/HTTP callback 形成 OOB 证据（盲漏洞专用，但许多其他类型也能改造） | 响应/后端两侧都拿不到证据 |
| ⑦ 回灌针对性 payload | 基于②③⑤⑥的发现，构造针对性 payload 再走一遍黑灰盒（如绕过特定校验、命中真实落点路径、附带回连标记） | 完成上述任意步骤后 |

### 4.2 按攻击类型推荐升级路径

| 攻击类型 | 优先证据来源 | 升级顺序 |
|---|---|---|
| 路径穿越（读取） | B 响应自证 | 自证 → ② 白盒读过滤 → ⑦ 针对性回灌 |
| 路径穿越（写入）| A 后端通道 | ④ → ③ → ⑤（监控瞬态文件）→ ⑥ 带外触发 → ⑦ 回灌 |
| Zip Slip | A 后端通道 | 业务接口列表 → `myssh` 全目录扫 → ⑤ 监控瞬态 → ③ 白盒看解压实现 |
| SQL 注入（回显） | B 响应自证 | 自证 → ③ 白盒读 SQL 模板 → ⑦ 针对性 payload |
| SQL 注入（盲） | A 带外回连 | 布尔/时间盲（中）→ ⑥ OOB（已确认）|
| SSRF | A 带外回连 | 直回显 B → ⑥ OOB → ③ 白盒读出网代码 |
| 命令注入 | A 带外回连 | 回显 B → ⑥ OOB → ③ 白盒读 exec 调用 |
| XSS | B 响应自证 | 反射型自证 → 存储型走业务接口验证 |
| 越权 | B 响应自证 | 返回他人可核对数据 → ③ 白盒读权限校验 |

### 4.3 终止条件

下最终结论需满足以下任一：

- **达到「已确认」**：置信度升级到「成立 + 已确认」或「不成立 + 已确认」，直接落盘结论。
- **合理路径已穷尽**：与本攻击类型相关的 4.1 步骤（参照 4.2 推荐顺序）**已全部尝试**，仍无法到达「已确认」 → 按当前最高证据下结论：「不成立 + 中」或「不确定」。报告必须列出"升级路径记录"段，写明走过的每一步及其结果。

> "升级路径记录"段是结论不被推翻的依据——审视者据此判断"是不是该试的都试了"。
