# 用例库使用规范

本目录用于存放**业务上下文卡片**。每个用例描述一个被测目标独有的业务逻辑、操作约束、接口时序等"业务怪癖"，让 skill 在测试时能避开业务雷区。

## 用例 ≠ 执行脚本

- 用例**不规定**要测试什么漏洞、用什么 Payload、打什么攻击模式。
- 攻击模式的匹配仍由 `攻击模式特征/` + `核心策略/` 动态决定。
- 用例只负责：**让 skill 知道这个目标是什么样子的**。

## 目录结构

```
用例库/
├── 说明.md                        # 本文件
├── <业务用例名A>/
│   ├── 说明.md                    # 必需，用例主文件
│   └── <其他物料>                 # 可选：样本数据、期望响应、示意图等
└── <业务用例名B>/
    └── 说明.md
```

- 用例目录名使用简洁的中文业务名，与 `说明.md` frontmatter 中的 `name` 保持一致。
- 物料文件（样本、期望响应等）自由命名，与 `说明.md` 同目录。

## `说明.md` 格式

```markdown
---
name: <与目录名一致的业务名>
description: <一句话概括>
requires:                           # 可选，仅有强前置依赖时才写
  - 子功能指引/<xxx>/执行步骤.md
---

## 业务特殊性
（描述这个业务不同于一般业务的地方：特殊字段、会话绑定、异步回调、前后端约定、隐式逻辑等）

## 操作约束
（描述测试时必须遵守的节流、频控、清理、账号要求等）

## 相关接口与字段
（列出涉及的接口、关键字段）

## 备注
（任何其他需要让 skill 知道的上下文）
```

## frontmatter 字段

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 是 | 唯一标识，任务中按此名引用 |
| `description` | 否 | 一句话概括用例 |
| `requires` | 否 | 仅当用例有明确前置依赖（如"必须先读子功能指引"）时填；大多数用例可缺省 |

## 正文写作原则

- **写业务，不写攻击**：描述"后端会把 manifest 里的 name 字段拼接到解压路径"，**不**写"这里可以做路径穿越"。
- **写约束，不写结论**：描述"连续失败 3 次后封禁 5 分钟"，**不**写"所以不能爆破"。
- **写已知事实，不写猜测**：如果某个字段是否可控不确定，留给测试阶段发现，不要在用例里预判。

## 引用方式

任务中按 `name` 引用，例如：

```
用例:
  - 上传业务配置ZIP
  - 带验证码的登录
```

写成数组按顺序执行；写成单个字符串只跑一个；写"全部"则按字典序遍历整个 `用例库/`。
