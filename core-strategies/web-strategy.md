# 安全测试流程：基于正常业务流量的攻击模拟分析

## 1. 前置准备
- **时间记录要求**  
  每次执行正常业务功能测试、攻击开始结束和关键操作节点，**必须**记录操作的开始时间与结束时间。  
  该时间范围可用于后续 `myproxy-query` 工具的精确查询，以准确获取对应请求记录。也可以通过该时间范围确认关键事件是否与操作相关，不在时间区间内的时间大概率是不相关的

## 2. 正常业务功能测试
- 按照标准用户流程完成一次完整的业务功能操作（例如登录、下单、提交表单等）。  
- 操作过程中**仅使用正常业务输入**，不包含任何攻击载荷。

## 3. 获取并追踪请求信息
- 调用 `myproxy-query` skill，基于步骤 1 记录的时间范围查询该业务操作所产生的 **HTTP 请求**。  
- 从查询结果中提取目标请求的 **唯一请求 ID**。  
- 使用 `trace_request` 工具，以获取的请求 ID 作为参数，获取该请求的**完整追踪信息**（涵盖后端处理日志、服务调用链、性能指标及其他关联上下文数据），用于支撑后续的深入分析。

## 4. 攻击模式识别
- 从项目根目录下的以下文件中读取并匹配攻击特征：
  - `攻击模式特征\Web页面特征.md`   → 用于识别页面层面的攻击模式
  - `攻击模式特征\HTTP请求特征.md` → 用于识别请求/响应层面的攻击模式
- 根据特征描述确定当前业务场景对应的潜在攻击模式及改造策略。

## 5. 构造并发送攻击请求
- **构造规则**：
  - **Header 保持原则**：除特殊情况外，**不得修改**原始请求的 HTTP 头部字段。
  - **Payload 修改原则**：仅对请求体（Body）进行最小必要改动以注入攻击载荷。

- **发送工具选择原则**：
  | 场景类型 | 推荐工具 | 说明 |
  |----------|----------|------|
  | 发送简单的独立 HTTP 请求 | `python requests` 库 或 `curl` 命令行工具 | 仅需发送一次请求并获取响应，无浏览器上下文依赖 |
  | 涉及复杂浏览器交互的测试场景 | Chrome DevTools MCP 或其他浏览器自动化工具 | 例如需模拟用户点击、处理重定向链、执行 JavaScript 代码等复杂流程 |

  **注意**：若测试目标仅为发送一个已构造好的攻击请求，不应使用浏览器自动化工具，以避免引入不必要的环境变量和性能开销。

## 6. 输出与记录
- 记录发送的攻击请求完整内容（URL、方法、Headers、Body）。  
- 记录服务端响应状态及关键响应内容。  
- 结合 `trace_request` 获取的完整追踪信息进行攻击效果分析与问题定位。
