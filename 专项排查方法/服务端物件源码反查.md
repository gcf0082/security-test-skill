# 服务端物件源码反查

> 适用：已经在后台访问到某个具体物件（文件 / 配置 / 二进制 / 内存中的字符串等），需要从 `项目代码/` 反查其在代码中如何被生成、读取、引用、暴露——以判断它是否构成安全风险或异常状态的来源。

## 1. 输入物件信息（用户提供 / 现场记录）

- 绝对路径或相对路径片段（如 `/var/log/app/login.log`、`upload/shell.jsp`、`secrets.yml`）
- 关键属性：mtime / size / 文件头 / 进程 owner（取决于是什么物件）
- 发现来源（哪条 `myssh` 命令 / 哪个日志查到的）

## 2. 排查流程

### 步骤一：在源码中精确搜索物件标识

按物件类型挑搜索锚点（按命中精度递减选择）：

- **文件名 / 路径片段**：`grep -rn '<filename>' 项目代码/` —— 同时命中字符串字面量、配置文件、注释。
- **路径常量**：找代码里的路径拼接（部署根 + 相对路径）；某些项目用 `application.yml` / `config.properties` 占位，先解析配置中间层再回到代码。
- **业务关键字**：物件内容里的特征字段名（如 `ak`、`secret_key`、自定义 marker），用于反查格式化 / 序列化代码。

### 步骤二：分类引用点

对每个命中按"角色"归类：

- **生成点**：代码主动写入这个文件 / 落盘这条日志的位置（`Files.write` / `LoggerFactory` / `FileOutputStream` 等）
- **读取点**：代码主动读这个文件的位置（`Files.read` / `Properties.load` / `YamlReader` 等）
- **引用点**：代码引用了路径但未直接 I/O（参数传递、注解 default value、文档字符串）

### 步骤三：每个引用点做白盒判定

- **生成点**：物件内容是否含外部用户输入？是否含敏感字段（凭据 / PII）？写入是否带脱敏？
- **读取点**：读取后流向哪里——回显给客户端、注入命令、解析后再写库？
- **引用点**：是否会被路由覆盖（如静态资源服务）？是否在错误响应中被暴露？

判定时套用 `攻击模式库/白盒攻击模式/通用白盒测试原则.md` §通用流程的"高危操作枚举 / 输入溯源 / 校验充分性"——本流程是它的"反向触发版"（先有产物，再找代码）。

### 步骤四：必要时回灌灰盒 / 黑灰盒

若代码分析发现物件可被外部用户触发或读取 → 按对应攻击模式的 `攻击模式库/黑灰盒攻击模式/<X>/测试方法.md` 派发 subagent 验证。

## 3. 输出契约（落盘 `报告/`）

| 字段 | 说明 |
|---|---|
| 物件信息 | 路径 / 大小 / mtime / 发现来源 |
| 源码引用清单 | 每条 `<文件>:<行号> [生成 / 读取 / 引用]` + 5~20 行上下文 |
| 用途判定 | 一段话说明物件由代码哪条路径生成、被哪条路径使用 |
| 风险等级 | **高 / 中 / 低 / 无风险**（参照 `通用白盒测试原则.md` 的判定） |
| 复测建议 | 是否需进一步派发灰盒 / 黑灰盒 subagent 验证 |

## 4. 红线

- ❌ 不直接修改 / 删除发现的物件——保留现场。
- ❌ 不在源码搜索时打开 binary 文件做内容修改；只读分析。
- ❌ 物件含敏感凭据时，报告中以脱敏样例引用，不复制原值（参照 `攻击模式库/黑灰盒攻击模式/敏感信息/测试方法.md` §4 红线）。
- ❌ 不把这条流程当作"常规白盒"的替代——它是用户**显式触发**的专项；常规模式分派下不进本流程。
