### 点击劫持攻击模式测试策略

> 🔒 **成立判定门槛**：关键操作页面**响应头未设置** `X-Frame-Options` 或 CSP `frame-ancestors`，且本地构造 PoC iframe 能成功嵌入该页面，即自证。通用判定 / 置信度见 `通用黑灰盒测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. 探测请求构造（一次到位）

列出关键操作页面（修改密码、转账、绑定手机、授权同意页等），逐个：

- 直接 `curl -I <URL>` 或浏览器 DevTools 取响应头，看是否设置 `X-Frame-Options` / `Content-Security-Policy: frame-ancestors`。
- 头未设置 → 本地构造一个 PoC iframe HTML（仅本机文件系统打开，不部署到外网），看能否真正嵌入页面：
  ```html
  <iframe src="<目标 URL>" width="800" height="600"></iframe>
  ```

只本地验证一次，不发到任何外网域。

如果走的是浏览器请求，按 `通用黑灰盒测试原则.md` §四"后备路径——时间范围查询"用 `nowTime.bat` 记前后时间戳；如果直接 `curl` 就用响应头中的 `X-Request-Id`。

#### 2. 响应观察（直接证据）

| 响应/PoC 表现 | 可能含义 | 结论 |
|---|---|---|
| `X-Frame-Options: DENY` / `SAMEORIGIN` 或 CSP 含 `frame-ancestors 'self'` | 已设置防护 | "无问题" |
| 两类头**都没有** + PoC iframe 能嵌入显示 | 点击劫持成立（自证） | "确认" |
| 头未设置但 PoC 中页面拒绝渲染（业务自带 JS frame-busting 但易被绕过） | 弱防御 | 算"中"——升级看 frame-busting 实现 |

#### 3. 后端核验 / 升级

- **响应头本身是直接证据**——头未设置 + PoC 能嵌 → "确认"。
- **结论不明**（如有 JS frame-busting）→ 置信度"中" → 转白盒读响应头中间件 / 安全策略配置（关键词 `X-Frame-Options` / `frame-ancestors` / `Helmet` / `SecurityHeadersFilter`），确认头为何未下发，并评估 JS frame-busting 是否能被 `sandbox` 属性绕过。

#### 4. 红线

- ❌ PoC iframe **仅本机构造验证**，不部署到任何可被他人访问的域。
- ❌ 不在测试中触发关键操作（点击转账按钮等）；只验证"页面能被嵌入"即可。
- ❌ 响应头未设置不等于直接成立——必须 PoC 能真嵌进去；有些页面靠 CSP `default-src` 间接挡住。
