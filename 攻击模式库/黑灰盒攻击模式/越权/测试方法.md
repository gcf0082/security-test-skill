### 越权攻击模式测试策略

> 🔒 **成立判定门槛**：必须有"低权限账号成功访问/修改高权限账号资源"的直接证据——B 账号调 A 账号的资源接口返回 A 的真实数据（自证）；或修改类操作改完后切回 A 账号查得到改动。通用判定 / 置信度见 `通用测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. 探测请求构造（一次一组，不堆）

准备**两个真实账号 A、B**，分别登录后抓基线请求。横向越权测一次：

- 用 **B 的会话**（cookie / token），调一个原本属于 **A 的资源**接口（替换 URL 路径中的资源 ID、或 body 中的资源/用户 ID 字段）。
- 其他请求字段逐字保持 B 抓到的样子。

纵向越权（普通用户 → 管理员）同理：用普通账号会话调管理接口路径。

> 不要枚举大量 ID 值——发一个 A 的真实 ID 即可判通道。结论不明转白盒读权限校验代码。

发送后**立即记录响应头 `X-Request-Id`**。

#### 2. 响应观察（线索，不是判定）

| 响应表现 | 可能含义 | 不能直接下的结论 |
|---|---|---|
| 200 + 回显 A 的真实数据（用 A 账号视角对照一致） | 越权成立（自证） | "确认" |
| 200 + 数据为空 / 仅返回 B 自己的数据 | 后端按会话强制过滤了 | "无问题"——可能查询条件已注入 user_id |
| 403 / 401 / "no permission" | 鉴权命中 | "无问题"——记"防御观察" |
| 200 但返回结构异常（部分脱敏 / 缺字段） | 仅部分字段越权 | 算"中"，按字段差异决定升级 |

> "数据是不是 A 的"判定标准：以 A 账号本视角调同接口的返回为基线对比；不要拿响应字段名瞎猜。

#### 3. 后端核验 / 升级

- **响应自证已足够**：B 视角拿到 A 数据 → "确认"。
- **http_req_query 反查**：用 `X-Request-Id` 拿后端处理日志中的鉴权决策（`AuthInterceptor` / `@PreAuthorize` 命中情况、SQL where 条件是否注入了 user_id）。
- **结论不明** → 置信度"中" → 转白盒读权限校验代码（关键词 `@PreAuthorize` / `@Secured` / 拦截器 / `getCurrentUser` / SQL where 子句是否注入会话用户），定位是否真的在每个接口都有鉴权，再回灌一次（可能换不同接口）。

#### 4. 红线

- ❌ 不批量 ID 枚举；发一两个真实 ID 即可判通道。
- ❌ 不在测试中执行**写类越权**到真实数据（修改/删除真实订单、改他人密码等）——除非用户授权，且优先选可回滚的资源。
- ❌ 响应"成功"不直接判漏洞成立；响应"失败"不直接判无问题。
