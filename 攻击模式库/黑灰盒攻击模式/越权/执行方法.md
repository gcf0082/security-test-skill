### 越权测试策略

越权分为**水平越权**（同角色跨用户）与**垂直越权**（低权限访问高权限接口）。

#### 1. 识别条件
- 请求参数或路径中含有**资源 ID**（`orderId`、`fileId`、`userId`、`docId`）。
- 请求体中出现**用户身份字段**（`uid`、`targetUserId`、`accountId`）。
- 管理后台 URL 路径（`/admin/`、`/manage/`）。
- 接口返回数据中含有他人信息的字段（头像、手机号、地址）。

#### 2. 测试方法

**水平越权**：
1. 使用账号 A 登录，操作某资源，记录其 ID（如 `orderId=10001`）。
2. 保留 A 的完整会话（Cookie/Token 不变）。
3. 将请求中的资源 ID 替换为账号 B 的资源 ID（如 `orderId=10002`）。
4. 观察响应：
   - **有漏洞**：返回 B 的订单详情 / 可操作 B 的资源。
   - **无漏洞**：返回 403/404 或统一错误。

**垂直越权**：
1. 使用**普通用户 A** 登录，获取会话。
2. 已知某管理接口（从管理员抓包或 JS 文件、Swagger 文档中获取）。
3. 用 A 的会话请求该管理接口（保持 URL、参数与管理员一致）。
4. 观察是否成功执行。

**附加技巧**：
- **参数增补**：普通接口请求中追加 `isAdmin=true`、`role=admin`、`userId=<他人>` 等字段，部分后端会直接采信请求中的字段。
- **HTTP 方法切换**：`GET` 禁用时尝试 `POST`/`PUT`/`DELETE`，反之亦然。
- **Referer/来源头去除**：部分后端仅通过 Referer 判断是否来自管理页面。
- **ID 枚举**：若 ID 为自增整数，可快速枚举相邻 ID 探测归属。若为 UUID，则需通过其他接口/泄露获取。

#### 3. 构造原则
- **Header 保持原样**（尤其认证字段），**仅修改资源标识或身份字段**。
- 每次只改一个字段，避免多变量干扰。
- 准备**至少两个账号**：一个操作账号、一个目标账号；两者应为同级用户（测水平越权）或一普通一管理员（测垂直越权）。

#### 4. 判定依据
- 账号 A 能查询/修改/删除**不属于 A** 的资源。
- 普通用户能成功调用管理员专属接口（返回 200 且业务逻辑被执行，而非被前端路由拦截）。
- 响应中出现他人敏感字段（手机号、身份证号）。

#### 5. 常见误报与排除
- 有些资源确实是**公开**的（如公告、帮助文档），跨用户可读不算越权。
- 前端隐藏按钮不等于后端鉴权，需通过直接发包确认。
- 有些接口虽返回 200，但业务层返回 `{"code": "no_permission"}` 实际未执行——要看响应体业务码，不要只看 HTTP 状态码。
