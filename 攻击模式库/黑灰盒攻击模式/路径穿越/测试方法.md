### 路径穿越攻击模式测试策略

> 🔒 **成立判定门槛**：必须有"文件真实写入越界路径"的后端直接证据。响应只是线索。通用判定 / 置信度 / 核验通道见 `通用黑灰盒测试原则.md` 与 `核心策略/置信度与升级策略.md`，本文件不重复。

#### 1. Payload 构造（一次一个，不堆）

获取一次正常上传请求，**仅替换 filename 字段**，其他请求内容逐字保持：

- 正常请求文件名**未编码** → Payload 用 `../file.txt`（**单层 `../`** 即可——足以判断是否有防御）。
- 正常请求文件名**已 URL 编码** → 跟随原请求的编码层数（一层 `%2e%2e%2ffile.txt`；显式存在二次解码迹象时再用 `%252e%252e%252ffile.txt`）。

> 多层 `../`、`..%c0%af` / `....//` / `..\\` 等字符变体**本阶段不做**——留到白盒读过滤实现后针对性回灌。一次只发一个 Payload，目的是判断"有没有防御、防御到什么程度"，不是穷举绕过（依据 `通用黑灰盒测试原则.md` §一）。

发送后**立即记录响应头 `X-Request-Id`** 作为后端反查锚点（依据 `通用黑灰盒测试原则.md` §四）。

#### 2. 响应观察（线索，不是判定）

| 响应表现 | 可能含义 | 不能直接下的结论 |
|---|---|---|
| 200 / 201 / 返回上传句柄、业务无报错 | 受理成功，但可能字符已清洗后落到默认目录，或异步处理未完成 | "漏洞成立"——**仍需第 3 节核验落盘路径** |
| 4xx「invalid filename / illegal character」 | 前置过滤命中 | "无问题"——仍要查日志确认是否有部分写入痕迹 |
| 4xx / 5xx「FileNotFound / permission denied / cannot create /xxx/../file.txt」 | 路径已被解析到越界位置，仅写入阶段受阻 | "攻击失败"——错误栈往往直接暴露拼接后的真实路径 |
| 业务无错但返回的文件名被改写（`../` 被替换为空等） | 防御部分生效，文件以清洗后的名字落到默认目录 | 直接结案——记"防御观察"，仍走第 3 节核对默认目录是否真的是默认 |

无论成败，记录响应原文（状态码、Body 关键字段、`Location` / 自定义 header）作为后续分析素材。

#### 3. 后端核验（任一通道拿到直接证据即可）

- **A. 业务侧只读接口**：调用对应的状态查询 / 文件列表 / 详情接口，看返回的路径字段或列表中是否出现越界目录的条目。**异步处理场景必须轮询至终态**（成功/失败）后再核验，不能在中间态判定。具体接口路径以被测项目为准。
- **B. `http_req_query` 反查事件流**：用步骤 1 记录的 `X-Request-Id` 拿请求/响应、后端处理日志、调用链。重点看：
  - 业务日志的**最终落盘路径**字段；
  - 错误栈中是否含越界后的真实路径片段；
  - 调用链中文件 I/O 节点（最贴近落盘的那一层）的入参。
- **C. `myssh` 直查文件系统**（环境允许时）：发送请求**前**用 `nowTime.bat` 记基线时间戳，事后跑 `find <目标目录> -newer <基线> -ls` 比对越界目录是否出现新增文件；目标目录与默认上传目录以被测项目实际为准。

任一通道直接证明「文件落到越界路径」 → 置信度 **确认**。
仅响应可疑、无后端证据 → 置信度 **中** —— 按 `核心策略/主策略.md` 转白盒读过滤 / 路径拼接代码理解防御边界，再回灌一次有针对性的 Payload。

#### 4. 红线（再强调一次）

- ❌ 不堆多层 `../` 或字符变体——一个代表性 Payload 即够。
- ❌ 响应"成功"不直接判漏洞成立；响应"失败"不直接判无问题——都要走第 3 节。
