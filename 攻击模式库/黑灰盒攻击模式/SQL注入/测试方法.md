### SQL注入攻击模式测试策略

> 🔒 **成立判定门槛**：必须有"用户输入被作为 SQL 语义执行"的直接证据——响应自证（错误堆栈含原始 SQL、Boolean 注入造成的可控差异、回显他人/系统数据）或后端 SQL 日志中拼出的语句被 Payload 改写。响应只是线索（自证除外）。通用判定 / 置信度见 `通用黑灰盒测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. Payload 构造（一次一个，不堆）

获取一次正常请求后，**仅修改可疑字段**（排序字段、ID、搜索关键词、分页参数等），其他逐字保持。三选一即可：

- **闭合探测**：`'`（最简单——直接看是否回显 SQL 异常）。
- **Boolean**：`' OR '1'='1`（看响应是否扩大返回集合或直接回显他人数据）。
- **盲注时延**：`' AND sleep(1)--`（仅当响应不回显且无错误时使用）。

正常请求若对参数做了 URL 编码 / Base64 / JSON-in-form 等包装，按相同格式套一层即可，不主动叠加变体。

发送后**立即记录响应头 `X-Request-Id`** 作为后端反查锚点。

#### 2. 响应观察（线索，不是判定）

| 响应表现 | 可能含义 | 不能直接下的结论 |
|---|---|---|
| 回显原始 SQL 异常（`SQLException`、`ORA-`、`syntax near`） | 拼接被打断，参数确实进了 SQL | 直接判"成立"——还要确认是否预编译失败而不是注入成功 |
| 返回集合数量异常变大、回显他人数据 | Boolean / UNION 类注入命中 | "未成立"——可能是业务过滤后的数据 |
| 时延 ≈ 注入的 sleep 时间（多次复现稳定） | 盲注通道存在 | 单次延迟≠注入；网络抖动也可能 |
| 200 + 业务正常回显，无错误 | 可能是参数化查询拦下了 | "无问题"——需 §3 看后端 SQL 日志 |

#### 3. 后端核验 / 升级

- **http_req_query 反查**：用 `X-Request-Id` 拿后端处理日志、SQL 执行链路。重点看**实际拼出的 SQL 语句**——Payload 字符是否原样进入 SQL，还是作为参数绑定。
- **响应自证已足够**：错误栈含原 SQL、Boolean 差异稳定复现、UNION 回显他人数据——任一即"确认"。
- **结论不明** → 置信度"中" → 转白盒读 ORM 调用 / DAO 层拼接代码（关键词 `Statement` / `concat` / `${}` / `prepareStatement`），定位是否真的预编译，再回灌一次有针对性的 Payload。

#### 4. 红线

- ❌ 不堆 `union select 1,2,3...,n` 列数枚举、不做"sqlmap 风格"穷举绕过——结论不明就读代码。
- ❌ 时延"看上去像"不算证据，至少复现 2 次且差异显著（>3× 基线）。
- ❌ 响应"成功"不直接判漏洞成立；响应"失败"不直接判无问题。
