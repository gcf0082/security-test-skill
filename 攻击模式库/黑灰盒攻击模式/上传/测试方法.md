### 上传攻击模式测试策略

> 🔒 **成立判定门槛（后端核验）**：上传类漏洞的"成立"定义是文件真实写入了越界路径，通用判定原则见 `攻击模式库/黑灰盒攻击模式/通用测试原则.md`、`踩坑提醒.md`。上传场景各通道的具体操作：
>
> - **业务侧只读接口**：调用上传/导入相关的状态查询、文件列表、详情等只读接口读后端文件落点。**异步解压/异步处理场景必须轮询至处理终态（成功/失败）后再核验**，不能在中间态判定。（示例：`GET /<导入接口>/<id>` 返回的路径字段、`GET /<列表接口>` 出现预期目录之外的条目；具体接口路径以被测项目为准）
> - **SSH 直查**：若环境允许 `myssh`，跑只读命令观测目标目录下时间窗内新增/异常文件。（示例：`find <目标目录> -newer <基线时间戳> -ls`；目录与基线以被测项目实际为准）
> - **带外回连**：适用于上传后触发服务端解析文件的场景（如 XML / SVG / XXE / SSRF 解析链）。

#### 1. 普通文件上传 - 文件名路径穿越

获取正常上传请求后，若包含文件名参数：

- **前置判断**：观察正常请求中文件名是否已被 URL 编码（出现 `%2e%2e%2f` 等），以此决定 Payload 构造方式。
- **测试方法**：
  - 未编码时：直接使用 `../file.txt`（优先单层 `../`）。
  - 已编码时：使用对应编码形式（如 `%2e%2e%2ffile.txt` 或 `%252e%252e%252ffile.txt`）。
- **原则**：Payload 装入 `filename` 字段（multipart 部分头的 `filename=` 或 JSON 参数值），仅修改该字段，其他请求内容保持不变；**每种编码形式只试 1~2 个代表性 Payload**，结论不明转白盒读过滤代码，不要堆 Payload。
- **分析重点**：响应只是线索，不是判定依据。必须通过上方「成立判定门槛」中的通道 A/B/C 之一核验服务端实际文件落点；未取到后端证据 → 判「不确定」，转白盒读过滤代码后回灌。

#### 2. 压缩包上传 - Zip Slip 测试

当业务支持上传 ZIP 文件时：

- **测试方法**：构造内部包含 `../file.txt`、`../../file.txt` 等穿越路径的 ZIP 包。
  > **注意**：这里的 `../` 是 ZIP 内部的**条目名**（记在 ZIP 元数据的路径字符串里）；越界要等目标服务端解压该 ZIP 时才发生，所以构造好 ZIP 后必须把它作为 multipart 上传发给目标接口。
- **原则**：优先单层穿越，保持压缩包结构和请求其他部分不变；**本类测试只构造 1~2 个代表性 ZIP**，结论不明转白盒看解压实现，不要堆变体。
- **分析重点**：响应返回上传/导入句柄仅表示上传受理，不代表解压成功或路径越界。必须等异步解压完成（轮询业务返回的导入句柄至处理终态；具体接口与字段以被测项目为准），再通过通道 A/B/C 核验实际解压落点；未取到后端证据 → 判「不确定」，转白盒看解压实现后回灌。
