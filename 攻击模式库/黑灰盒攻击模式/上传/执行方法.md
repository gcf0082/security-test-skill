### 上传攻击模式测试策略

> ⚠️ **关键原则（路径穿越测试方向）**：所有 `../` 的测试本质是把 `../` **装进发向远程的 payload**（filename 参数值、ZIP 内部条目名、表单字段、请求 body 等），让**远程服务端**在保存/解压/写盘时越界——这是对远端的攻击。
>
> **严禁**在本地 shell 里操作 `../`（如 `ls ../`、`cat ../etc/passwd`、`cp foo ../bar`、把文件保存到本地 `../xxx` 等）——本地文件系统操作既不构成攻击，也不产生任何测试证据。
>
> 判断口径：所有 `../` 必须能在**发往远端的 HTTP 请求**中被抓包/日志证实（body、字段值、上传文件内部结构），并让远程服务器处理后产生异常落点。

> 📌 **尝试次数克制**：每种上传测试类型（文件名穿越、Zip Slip、扩展名绕过等）只尝试 **1~2 个代表性 Payload**——目的是判断"有没有过滤、过滤到什么程度"，**不是穷举绕过**。
> 结论不明时按 `核心策略/主策略.md` 的"黑灰盒主导 + 白盒按需补充"走白盒读过滤代码，再用有针对性的 Payload 回灌黑灰盒验证——**禁止在黑灰盒里堆 Payload**。

#### 1. 普通文件上传 - 文件名路径穿越

获取正常上传请求后，若包含文件名参数：

- **前置判断**：观察正常请求中文件名是否已被 URL 编码（出现 `%2e%2e%2f` 等），以此决定 Payload 构造方式。
- **测试方法**：
  - 未编码时：直接使用 `../file.txt`（优先单层 `../`）。
  - 已编码时：使用对应编码形式（如 `%2e%2e%2ffile.txt` 或 `%252e%252e%252ffile.txt`）。
- **原则**：`../` 必须出现在**发往服务端的 `filename` 字段值**里（multipart 部分头的 `filename=` 或 JSON 参数值），而不是本地文件路径；仅修改该字段，其他请求内容保持不变；**每种编码形式只试 1~2 个代表性 Payload**，结论不明转白盒读过滤代码，不要堆 Payload。
- **分析重点**：观察响应及文件实际存储位置，判断是否穿越成功。

#### 2. 压缩包上传 - Zip Slip 测试

当业务支持上传 ZIP 文件时：

- **测试方法**：构造内部包含 `../file.txt`、`../../file.txt` 等穿越路径的 ZIP 包。
  > **注意**：这里的 `../` 是 ZIP 内部的**条目名**（记在 ZIP 元数据的路径字符串里），不是把文件写到本地 `../` 目录。构造好 ZIP 后，**要把该 zip 作为 multipart 上传发给目标接口**才是真正的攻击动作；`../` 越界要等目标服务端解压时才发生。
- **原则**：优先单层穿越，保持压缩包结构和请求其他部分不变；**本类测试只构造 1~2 个代表性 ZIP**，结论不明转白盒看解压实现，不要堆变体。
- **分析重点**：检查解压后文件是否落到预期目录之外。
