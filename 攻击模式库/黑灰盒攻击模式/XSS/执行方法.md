### XSS 测试策略

#### 1. 识别条件
- 备注、昵称、签名、评论、富文本编辑器、邮件内容、系统通知、工单处理意见等**回显型**输入字段。
- URL 参数、搜索关键词、错误提示等会被**原样拼接**到页面的位置（反射型）。
- 前端 JavaScript 中的 `innerHTML`、`document.write`、`eval`、`location.hash`、`postMessage` 消费点（DOM 型）。

#### 2. 测试方法
先探测再武器化，按上下文区分：

**阶段一：探测可回显字符**
- Payload：`'"<xss>`
- 查看响应与渲染后 DOM 中这五个字符的保留/转义情况，以确定上下文类型。

**阶段二：按上下文构造**

| 上下文 | 典型 Payload |
|--------|-------------|
| HTML 标签之间 | `<svg onload=alert(1)>`、`<img src=x onerror=alert(1)>` |
| HTML 属性值中（双引号闭合） | `" autofocus onfocus=alert(1) x="` |
| HTML 属性值中（无引号） | ` onmouseover=alert(1) ` |
| `<script>` 内字符串 | `';alert(1);//` |
| URL/`href`/`src` 中 | `javascript:alert(1)` |
| JSON 被 `innerHTML` 渲染 | `</script><svg onload=alert(1)>` |

**阶段三：存储型跟踪**
- 提交后切换视图（如管理员后台、他人可见列表），验证 Payload 是否在其他用户上下文执行。
- 对富文本编辑器，特别注意 `javascript:` 伪协议、`srcdoc`、`onerror`、`style=expression()` 是否被过滤。

#### 3. 构造原则
- 使用**无害 Payload**：`alert(document.domain)` 或在 `alert` 内加唯一标识 `alert('XSS-TEST-<uuid>')`，便于在大量日志中定位。
- 每个字段**独立测试**，不要同时投多个 Payload。
- 优先使用**不依赖用户交互**的事件（`onload`、`onerror`、`onfocus autofocus`），排除纯理论漏洞。

#### 4. 判定依据
- 浏览器弹出 `alert` 或控制台打印标识。
- 响应 HTML 中 Payload **未被转义**（`<` 未变 `&lt;`）。
- DOM 中观察到新的 `<script>` 节点或事件处理器属性。

#### 5. 常见误报与排除
- 响应体里保留了 `<script>` 但 `Content-Type: text/plain` 或下载响应，不会被执行。
- CSP 严格的页面，即使注入成功也无法执行内联 JS——需评估是否可以通过白名单域上传 JS 再引用。
- 仅在自己账号下回显不构成越权型 XSS，需要证明能影响**其他用户**。
