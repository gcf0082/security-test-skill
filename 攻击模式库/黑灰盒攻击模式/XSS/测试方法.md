### XSS攻击模式测试策略

> 🔒 **成立判定门槛**：必须有"用户输入在响应中被作为可执行 HTML/JS 渲染"的直接证据——在浏览器（或 headless）打开命中页面后实际触发脚本执行，或 payload 在响应中位于可执行上下文且未被编码，即自证。通用判定 / 置信度见 `通用测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. Payload 构造（一次一个，不堆）

先看正常响应中**可疑字段在 HTML 的什么位置**（HTML 文本 / 标签属性 / JS 字符串 / URL 上下文），按上下文选**一种**：

- HTML 文本：`<svg/onload=alert("marker_<随机串>")>`
- 属性内：`" autofocus onfocus=alert("marker_<随机串>") x="`
- JS 字符串内：`";alert("marker_<随机串>");//`

> 上下文还没看清就先发 HTML 文本那条；结论不明转白盒读输出编码代码，不要堆变体。

发送后**立即记录响应头 `X-Request-Id`**。

#### 2. 响应观察（线索，不是判定）

| 响应表现 | 可能含义 | 不能直接下的结论 |
|---|---|---|
| Payload 在响应中**原样出现**且位于可执行上下文 | 接近成立，但要在浏览器/headless 里触发一次确认 | 直接落"确认"——浏览器没真触发不算 |
| Payload 被 HTML 实体编码（`&lt;`、`&quot;`） | 输出端转义生效 | "无问题" |
| Payload 被截断或字符过滤 | 防御部分生效 | 记"防御观察"，看是否能改上下文绕开 |
| 响应不回显 | 可能是存储型 → 找回显页面再观察 | "无问题"——需 §3 |

#### 3. 后端核验 / 升级

- **浏览器/headless 触发**：在干净环境（隔离浏览器/Playwright/Selenium）打开命中页面，看脚本是否真执行（`marker_<随机串>` 进 alert/console），即"确认"。
- **存储型**：找到回显该字段的所有页面（用户详情、列表、消息中心等），逐个看 §2 表。
- **myproxy-query 反查**：用 `X-Request-Id` 看后端写入路径，确认数据落盘的字段。
- **结论不明** → 置信度"中" → 转白盒读输出编码代码（关键词 `escape` / `htmlEncode` / `sanitize` / 模板的 `{{ var }}` vs `{{{ var }}}` / Vue `v-html` / React `dangerouslySetInnerHTML`），再回灌一次上下文匹配的 Payload。

#### 4. 红线

- ❌ 不堆 payload 变体（事件大全 `onmouseover` / `onerror` / `onload` …）——一种上下文匹配的即可。
- ❌ 不发会改坏页面状态的 payload（`document.body.innerHTML=...`、清 localStorage 等）；用 `alert` 标识即可。
- ❌ 响应"原样回显"不等于成立——必须在浏览器里真触发或在白盒中确认上下文未编码。
