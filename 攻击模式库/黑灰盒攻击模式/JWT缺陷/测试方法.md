### JWT缺陷攻击模式测试策略

> 🔒 **成立判定门槛**：篡改后的 JWT 仍能通过校验、被服务端当作合法身份处理，即自证（具体表现：响应返回他人/管理员视角下的数据，或操作权限被识别为目标身份）。通用判定 / 置信度见 `通用黑灰盒测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. 探测请求构造（三选一，每次只测一种）

抓一个正常带 JWT 的请求（`Authorization: Bearer <token>` 或 Cookie）。**三选一**：

- **alg=none 攻击**：把 header 改为 `{"alg":"none","typ":"JWT"}`，签名段置空（保留两个点），payload 不变重发。
- **payload 篡改不重签**：解码 payload 改一个关键字段（如 `sub`、`role`、`uid`），签名段保持原值重发。
- **弱密钥**：用前 3 个最常见弱密钥（`secret`、`123456`、`<服务名>`）尝试本地重签，重发。

> 不要三种都堆；按上下文挑最可能的那种。无线索时按 ①、②、③ 顺序逐次试，但**每次仅一个 Payload**，结论不明转白盒。

每次记录响应头 `X-Request-Id`。

#### 2. 响应观察（线索，不是判定）

| 响应表现 | 可能含义 | 结论 |
|---|---|---|
| 200 + 返回篡改后身份对应的数据（如改 `sub` 为他人 ID 后回显他人数据） | 篡改被接受（自证） | "确认" |
| 401 / 403「invalid signature」 | 签名校验生效 | "无问题" |
| 401「token expired」/「invalid token」 | 校验生效 | "无问题" |
| 200 但数据仍是原账号 | 服务端可能取 cookie / session 而非 JWT | 算"中"——需 §3 判 JWT 是否真在用 |

#### 3. 后端核验 / 升级

- **响应自证已足够**：篡改 token 拿到他人/管理员数据 → "确认"。
- **http_req_query 反查**：用 `X-Request-Id` 拿 JWT 校验日志，看实际走的校验逻辑、是否真的验签。
- **结论不明** → 置信度"中" → 转白盒读 JWT 校验代码（关键词 `JWTVerifier` / `Jwts.parser` / `jsonwebtoken.verify` / `algorithm=` / `alg`），确认是否信任 header 里的 `alg`、是否硬编码 `HS256`、是否做过期校验。

#### 4. 红线

- ❌ 不跑弱密钥字典（`rockyou.txt` / `jwt_secrets.txt` 之类）；本阶段最多 3 个常见值。
- ❌ 不在测试中用拿到的伪造 token 执行**写类操作**（除非用户授权）；先验通道。
- ❌ 响应"401"不等于无问题——可能仅过期，需要看 `iat`/`exp` 是否被校验。
