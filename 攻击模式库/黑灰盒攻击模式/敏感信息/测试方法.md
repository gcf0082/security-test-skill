### 敏感信息攻击模式测试策略

> 🔒 **成立判定门槛**：响应中**直接含**敏感字段（密码、Token、API Key、SecretKey 原值；未脱敏的手机号 / 身份证 / 银行卡）；或错误响应栈暴露文件路径、框架版本、SQL 语句、内部 IP——任一即自证。通用判定 / 置信度见 `通用测试原则.md` 与 `核心策略/置信度与升级策略.md`。
>
> 白盒视角对应 `白盒攻击模式/敏感信息/测试方法.md`，本文件仅做黑灰盒侧的响应面观察。

#### 1. 探测请求构造（一次到位，多接口轮）

锁定几类**最可能回显敏感信息**的接口，每个发一次正常业务请求：

- **用户详情类**：当前用户、订单详情、地址详情、支付方式
- **管理后台类**（如有权限）：用户列表、配置查询、密钥管理
- **错误触发类**：故意构造非法参数（缺字段、错类型）让服务端报错，看错误栈

不要枚举所有 ID 或所有接口；选 3~5 个业务关键接口轮一遍即够。每次记录 `X-Request-Id`。

#### 2. 响应观察（直接证据为主）

| 响应表现 | 可能含义 | 结论 |
|---|---|---|
| JSON 字段含 `password` / `token` / `apiKey` / `secretKey` 原值 | 敏感凭据泄露（自证） | "确认" |
| `phone` / `idCard` / `bankCard` 未脱敏（完整 11 位 / 18 位 / 16+ 位） | 个人信息泄露（自证） | "确认" |
| 错误栈含文件路径（`/opt/app/...`）、框架版本（`Spring Boot 2.7.x`）、SQL 语句、内部 IP | 信息泄露（自证） | "确认" |
| 字段已脱敏（`13***1234`、`****@gmail.com`） | 防御生效 | "无问题" |
| 错误响应仅返回业务码 + 中性消息 | 错误处理收敛 | "无问题" |

字段名怀疑但无法肉眼判断是否真敏感时（如 `meta`、`extra`、`config`），再用 `X-Request-Id` 调 `http_req_query` 看完整响应体。

#### 3. 后端核验 / 升级

- **响应自证已足够**：字段值或错误栈本身就是判定证据。
- **http_req_query 反查**：用 `X-Request-Id` 拿后端处理日志，看是否还有更深层日志中的敏感信息泄露（请求/响应 dump、调用链中的明文密钥）。
- **结论不明** → 置信度"中" → 转白盒读响应序列化 / 脱敏代码（关键词 `@JsonIgnore` / `@JsonProperty` / `serializer` / `ResponseBody` / `mask` / `desensitize`），并参考 `白盒攻击模式/敏感信息/测试方法.md`。

#### 4. 红线

- ❌ 不批量枚举接口或 ID 抓敏感字段；3~5 个关键接口够判通道。
- ❌ 看到敏感字段后**只在报告中保留必要的、能证明问题的最小片段**（如脱敏后的样例 + 字段位置），不复制整段原值。
- ❌ 字段名疑似不等于真敏感——必须看到字段值或调试一次确认。
