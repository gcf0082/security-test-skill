### 敏感信息泄露测试策略

#### 1. 识别条件
- 用户列表、个人资料、订单详情、登录接口等可能返回用户敏感字段的响应。
- 响应 JSON 的字段数量明显多于前端需要（"字段冗余"）。
- 响应头/Cookie/错误页面中出现版本号、内部 IP、调试信息。

#### 2. 测试方法

**阶段一：接口响应字段审计**
对每一类主要接口，逐字段比对前端展示和响应数据：

| 风险字段 | 说明 |
|---------|------|
| `password`、`pwd`、`hash`、`salt` | 认证凭据（任意形式都属于高危） |
| `token`、`accessToken`、`refreshToken`、`sessionId` | 会话凭据 |
| `phone`、`mobile`、`idCard`、`身份证` | PII，应脱敏显示（`138****1234`） |
| `email`、`address`、`realName` | PII，需按场景判断是否应暴露 |
| `apiKey`、`secretKey`、`signKey` | 第三方密钥 |
| `internalIp`、`hostname`、`serverVersion` | 基础设施信息 |

**阶段二：列表接口越权**
- 许多用户列表接口默认返回所有用户的全量字段。
- 调用 `/api/users/list` 观察能否拿到他人的手机号、地址、注册时间。

**阶段三：错误信息泄露**
- 构造异常请求（畸形 JSON、错误参数、非法字符），观察响应：
  - 堆栈信息（包路径 → 技术栈）
  - SQL 错误（表名/字段名）
  - 文件路径（`/home/app/`、`C:\inetpub\`）
  - 框架版本（`Spring Boot 2.3.1`）

**阶段四：静态资源泄露**
访问常见敏感路径：
- `/.git/config`、`/.svn/entries`
- `/.env`、`/.env.production`、`/config.yml`、`/application.properties`
- `/phpinfo.php`、`/info.php`
- `/swagger-ui.html`、`/v2/api-docs`、`/actuator/env`
- `/WEB-INF/web.xml`、`/META-INF/`

**阶段五：JS/前端资源审计**
- 浏览器 DevTools → Sources，搜索 `apiKey`、`secret`、`password`、`internal`、`debug`。
- 审计前端路由配置，查找管理员路由/未启用功能接口。

#### 3. 构造原则
- 所有探测请求使用**正常会话凭据**，避免被统一拦截。
- 敏感路径访问**控制频率**，避免触发 WAF 封禁。

#### 4. 判定依据
- 响应中出现明文或可逆的凭据（密码、token、密钥）。
- 响应字段含非业务必要的 PII（未脱敏的手机号、身份证）。
- 错误堆栈包含可辅助其他攻击的路径/版本/SQL 信息。
- `/actuator/env`、`/.git/config` 等返回 200 且包含配置内容。

#### 5. 常见误报与排除
- 响应中返回 `id`、`createTime` 等并不算敏感。
- 返回的 `token` 若是**当前用户自己**的 token，不算泄露；关键是看是否返回**他人**的或后端内部密钥。
- Swagger 等调试接口在生产暴露本身是问题，但若访问需要认证并且只对开发人员开放，则风险降级。
