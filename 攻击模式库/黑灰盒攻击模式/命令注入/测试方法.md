### 命令注入攻击模式测试策略

> 🔒 **成立判定门槛**：必须有"用户输入被作为 shell 命令执行"的直接证据——响应回显命令输出（`uid=...`、文件内容）属自证；其他情况靠后端进程/日志中的 exec 调用记录。通用判定 / 置信度见 `通用测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. Payload 构造（一次一个，不堆）

获取一次正常请求，**仅修改可疑字段**（主机名、域名、IP、文件名等通常会被传给外部命令的字段），其他逐字保持。二选一：

- **回显式**：在原值后追加 ``$(id)`` 或 `` `id` ``——直接看响应是否含 `uid=` 标志。
- **盲打时延**：追加 `; sleep 5` 或 `| sleep 5`——仅当响应无回显时用。

> 多种分隔符（`;` / `&&` / `||` / `|` / 换行）择一即可。结论不明转白盒读命令调用代码，不要堆变体。

发送后**立即记录响应头 `X-Request-Id`** 作为后端反查锚点。

#### 2. 响应观察（线索，不是判定）

| 响应表现 | 可能含义 | 不能直接下的结论 |
|---|---|---|
| 响应含 `uid=`、`gid=`、文件内容等命令输出 | 命令注入成立（自证） | 直接落"确认" |
| 响应含 `sh: ... not found` / `command not found` 类错误 | 用户输入进了 shell，仅命令不存在 | "失败"——已经在 shell 里了 |
| 时延 ≈ sleep 时间（多次稳定复现） | 盲注通道存在 | 单次延迟≠注入 |
| 200 业务正常 + 无错误 | 可能输入被转义或调用 API 而非 shell | "无问题"——需 §3 |

#### 3. 后端核验 / 升级

- **myproxy-query 反查**：用 `X-Request-Id` 拿后端处理日志，看**调用链中外部命令节点**（`Runtime.exec` / `subprocess` / `os.system` 调用点）的入参，确认 Payload 是否进入命令字符串。
- **myssh 看进程历史**（环境允许时，仅只读）：在合理时间窗内看 `auditd` 日志或服务进程历史，避免直接 `bash` 历史这种宽口径。
- **响应自证**（`uid=` 回显、时延稳定）已足够 → "确认"。
- **结论不明** → 置信度"中" → 转白盒读外部命令调用代码（关键词 `Runtime.exec` / `subprocess.Popen` / `os.system` / `child_process.exec`）确认拼接位置与转义实现，再回灌一次。

#### 4. 红线

- ❌ 不堆分隔符变体（`;` / `&` / `|` / 反引号 / `$()` / 换行）——一种命中即可下结论或转白盒。
- ❌ 时延必须复现稳定（>3× 基线）才算证据。
- ❌ 响应"成功"不直接判漏洞成立；响应"失败"不直接判无问题。
