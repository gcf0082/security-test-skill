### 命令注入攻击模式测试策略

> 🔒 **成立判定门槛**：必须有"用户输入被作为 shell 命令执行"的直接证据——服务端按 Payload 创建出预期标识文件（首选证据）；或响应回显命令输出（`uid=...`、文件内容）属自证。通用判定 / 置信度见 `通用测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. Payload 构造（分阶段，不盲打）

获取一次正常请求，**仅修改可疑字段**（主机名、域名、IP、文件名等会被传给外部命令的字段），其他逐字保持。**按阶段推进，每阶段只在结论不明时才进入下一阶段**：

- **Phase A 单字符轻探测**：在原值后追加 `;` 或 `|` 中**任一个**，看响应。
- **Phase B 字符过滤边界扫描**（仅当 A 阶段响应明显报"非法字符 / invalid character / 禁止字符"等过滤命中时进入）：把命令注入的常见特殊字符各试一次——`;`、`&`、`&&`、`||`、`|`、反引号、`$()`、换行（`%0a`）、`${IFS}`——确认每一个是否都被过滤。**这是针对性的边界探测，不是盲打字典**；目的就是找出黑名单遗漏。
- **Phase C 文件创建 Payload**（A 直接通过 或 B 找到未被过滤字符后进入）：用未被过滤的分隔符构造 `<分隔符>touch /tmp/xygcf_<随机串>.txt`。**首选 `/tmp`**——默认所有进程可写，且核验时直接搜 `/tmp` 即可命中。

> sleep 时延（`; sleep 5`）和回显（`$(id)`）仅作兜底——前者受网络抖动影响、后者多数接口不回显。优先走文件创建。

每次发送后**立即记录响应头 `X-Request-Id`** 作为后端反查锚点。

#### 2. 响应观察（线索，不是判定）

| 响应表现 | 阶段含义 | 下一步 |
|---|---|---|
| 4xx / 5xx「invalid character」/「contains illegal char」/「禁止字符」 | A：字符过滤命中 | 进 Phase B 扫描所有特殊字符 |
| 200 + 业务正常无错误 | A 或 B 通道开放 | 进 Phase C 发文件创建 Payload |
| 响应回显 `uid=` / 文件内容 / 命令输出 | 命令注入成立（自证） | 直接落"确认" |
| 时延 ≈ sleep 时间（多次稳定复现 ≥2 次） | 兜底通道存在 | 仍走 §3 文件核验取直接证据 |
| C 阶段 200 + 无错误 | Payload 发出去了，未必执行 | §3 后端找文件 |

#### 3. 后端核验（找标识文件为主，深度监控为兜底）

- **首选 `/tmp` 直查**（环境允许 `myssh` 时）：`ls /tmp/xygcf_<随机串>.txt` 或 `find /tmp -name 'xygcf_*' -newer <基线>`。命中 → 命令注入"确认"。
- **若 Payload 未指定 `/tmp`**：按 `踩坑提醒.md §二` 收敛搜索范围——从 `http_req_query` 反查进程 / 用 `lsof` 看 cwd / `/proc/<pid>/cwd` 反推后端工作目录，再去那里找标识文件。
- **没找到 ≠ 失败**：换 `/tmp` 显式路径再发一次 Phase C；仍找不到再考虑下面的"深度监控"或转白盒。
- **http_req_query 反查**：用 `X-Request-Id` 拿后端处理日志，看调用链中外部命令节点（`Runtime.exec` / `subprocess` / `os.system`）的入参，确认 Payload 是否真的进入命令字符串。
- **深度监控通道（强先验下启用）**：当外部信号已强烈指向该接口极大概率会调外部命令——日志中有 "executing command…" 字样、白盒代码已读到 `Runtime.exec` / `subprocess.Popen` 调用点、相关文档明示、或用户强制要求——但上面通道仍拿不到直接证据时，在后台启动进程监控（`extrace` / `execsnoop` / `auditd` 监控 `execve` 等**只读工具，仅环境允许时**），开监控**之后**再发一次 Phase C Payload，观察时间窗内是否有可疑进程被启动、其命令行参数中是否含 Payload 字符串。命中即"确认"。监控完毕及时关闭，避免长期占用。
- **结论不明** → 置信度"中" → 转白盒读外部命令调用（关键词 `Runtime.exec` / `subprocess.Popen` / `os.system` / `child_process.exec`），确认拼接位置与转义/参数化实现，再回灌一次。

#### 4. 红线

- ❌ Phase B 仅在 A 阶段确认过滤后才做；A 直接通过就跳过 B，不做无谓字符扫描。
- ❌ 不发会改坏目标系统的 Payload——文件名仅 `/tmp/xygcf_<随机>.txt` 这种自带标识、可手动清理的形态；不要 `touch /etc/passwd`、不要 `rm -rf`。
- ❌ 不发 `sleep` 长时间或并发多发——时延做兜底证据时单次 `sleep 5` 足够，且要稳定复现 ≥2 次（>3× 基线）。
- ❌ 测试结束在报告"附带观察"或"清理项"里记录留下的标识文件，方便后续清理。
- ❌ 响应"成功"不直接判漏洞成立；响应"失败"不直接判无问题——都要走 §3。
