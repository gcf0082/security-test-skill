### CSV注入攻击模式测试策略

> 🔒 **成立判定门槛**：必须有"用户输入被原样写入导出文件单元格首字符且未做净化"的直接证据——下载导出文件后单元格首字符为 `=` / `+` / `-` / `@` / `\t` / `\r` 即自证。通用判定 / 置信度见 `通用测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. Payload 构造（一次一个，不堆）

在能影响导出内容的字段（昵称、备注、订单号、商品名、消息、地址等）填入**一个**最简公式：

- 优先：`=1+1`（无副作用，最干净的判定）
- 备选：`=cmd|'/c calc'!A1`（典型 DDE Payload，仅在 Excel 打开时触发；写入文件不会主动执行）

提交后**触发对应导出**（CSV / XLS / XLSX），下载导出文件。

发送提交请求与导出请求时**分别记录响应头 `X-Request-Id`**——前者锚定写入路径，后者锚定导出路径。

#### 2. 响应观察 / 文件观察（线索 + 直接证据）

| 文件表现 | 可能含义 | 结论 |
|---|---|---|
| 导出文件中目标单元格以 `=1+1` 等公式字符开头，原样写入 | 未做净化，CSV 注入成立（自证） | "确认" |
| 单元格内容前被加 `'`（单引号）、空格 或公式被转义 | 防御已生效（导出端净化） | "无问题"——记"防御观察" |
| 导出文件不含输入字段 | 该字段不在导出范围 | "无问题"——换字段或退出 |

注：用文本工具（`head`、`xlsx2csv`）查看，**不要在 Excel 里直接打开**含 DDE Payload 的文件。

#### 3. 后端核验 / 升级

- **响应自证已足够**：文件单元格首字符就是判定证据。
- **myproxy-query 反查**：用 `X-Request-Id` 拿导出请求的处理日志，看是否经过单元格净化中间件。
- **结论不明** → 置信度"中" → 转白盒读导出拼装代码（关键词 `CSVWriter` / `XSSFWorkbook` / `openpyxl` / `csv.writer` / `writeRow`），确认是否对首字符做了 `'` 前缀或公式字符过滤，再回灌一次。

#### 4. 红线

- ❌ 不堆公式变体（`@SUM(...)`、负号开头、Tab 起始等）——一种命中即可下结论。
- ❌ **不要在 Excel 里直接打开** Payload 文件——可能在测试机上触发命令。用文本工具看。
- ❌ 响应"成功"不直接判漏洞成立；响应"失败"不直接判无问题。
