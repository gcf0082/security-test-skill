### 二次认证攻击模式测试策略

> 🔒 **成立判定门槛**：高危操作（修改密码、删除账户、转账、解绑设备等）在已登录态下**不要求重新输入密码或二因子**即可成功，且后端确实执行了该操作。通用判定 / 置信度见 `通用测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. 探测请求构造（一次一接口）

锁定一个高危操作接口，**保持已登录会话**（cookie / token），直接发起请求；不带任何二次认证字段（`password`、`smsCode`、`totp` 等）：

- 完整重放抓到的高危操作请求，**仅删除二次认证字段**（或保留字段但置空），其他逐字保持。

只测一种最关键的操作（业务最敏感的那个，通常是改密 / 转账）；多个接口轮一遍而不是堆 payload。

发送后**立即记录响应头 `X-Request-Id`**。

#### 2. 响应观察（线索，不是判定）

| 响应表现 | 可能含义 | 不能直接下的结论 |
|---|---|---|
| 200 + 操作成功 + 后端状态变更（密码改了 / 转账记录有了） | 二次认证缺失（自证） | "确认" |
| 4xx「password required」/「verification code required」 | 二次认证生效 | "无问题"——记"防御观察" |
| 200 但操作未真正生效 | 可能伪成功 | 算"中"——必须 §3 验状态 |
| 跳到二次认证页面 | 防御生效 | "无问题" |

#### 3. 后端核验 / 升级

- **业务侧只读接口验状态**：调用对应资源详情接口看是否真改了（密码改了 → 用新密码登录；转账成功 → 看流水）。**这是必须的一步**——不能仅凭 200 判成立。
- **http_req_query 反查**：用 `X-Request-Id` 拿后端处理日志，看请求是否经过二次认证中间件、是否落到业务逻辑写库。
- **结论不明** → 置信度"中" → 转白盒读关键操作鉴权流（关键词 `@PreAuthorize` / 拦截器 / `verifyPassword` / `verifyTOTP`），确认是否真的检查了二次认证。

#### 4. 红线

- ❌ 不要真改用户密码 / 真转账——除非用户授权且选用可恢复的测试资源（沙箱账户、预先备好的测试余额）。
- ❌ 不批量轮所有接口；先选最敏感的那个验通道，结论清楚后再扩展。
- ❌ 响应 200 不等于真生效——必须独立通道核验状态变更。
