### SSTI攻击模式测试策略

> 🔒 **成立判定门槛**：必须有"用户输入被作为模板表达式渲染"的直接证据——响应回显计算结果（如 `49`）属自证；模板引擎错误栈暴露引擎名/版本是辅证。通用判定 / 置信度见 `通用黑灰盒测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. Payload 构造（一次一个，不堆）

获取一次正常请求，**仅修改可疑字段**（自定义模板内容、邮件标题、报表字段、消息模板等），其他逐字保持。先观察响应或前端线索判断模板引擎类型，再三选一：

- **Jinja2 / Twig / Liquid**：`{{7*7}}`
- **FreeMarker / Thymeleaf**：`${7*7}`
- **EJS / ERB / JSP EL**：`<%= 7*7 %>` 或 `${7*7}`

> 还不知道引擎时只发一种最常见的（`{{7*7}}`），看响应里是否被渲染或暴露引擎错误。结论不明就读代码而不是堆变体。

发送后**立即记录响应头 `X-Request-Id`** 作为后端反查锚点。

#### 2. 响应观察（线索，不是判定）

| 响应表现 | 可能含义 | 不能直接下的结论 |
|---|---|---|
| 响应中出现 `49` 渲染结果 | SSTI 成立（自证） | 直接落"确认" |
| 响应原样回显 `{{7*7}}` 字符串 | 模板未渲染或字段被作为字面量 | "无问题"——需 §3 看是否其他模板路径 |
| 响应含模板引擎错误（`FreemarkerException`、`Jinja2 traceback`、`UndefinedError`） | 输入进了模板解析 | 接近"确认"，按错误栈定位引擎/版本，回灌一次 |
| 200 + 业务正常 | 可能字段未走模板渲染 | "无问题"——需 §3 |

#### 3. 后端核验 / 升级

- **响应自证已足够**：渲染结果回显或引擎错误栈即可下结论。
- **http_req_query 反查**：用 `X-Request-Id` 拿后端处理日志，看是否触发模板渲染调用。
- **结论不明** → 置信度"中" → 转白盒读模板渲染调用点（关键词 `Template` / `render` / `process` / `Velocity` / `Freemarker` / `Jinja`）确认是否拼接用户输入到模板字符串，再针对识别出的引擎回灌一次。

#### 4. 红线

- ❌ 不堆多种引擎语法 Payload；先看响应/代码线索判断引擎，再针对性发一次。
- ❌ 不在黑灰盒阶段直接发 RCE Payload（`{{config.__class__.__init__.__globals__...}}` 之类）——确认 SSTI 后再升级，且仅在用户授权范围内。
- ❌ 响应"成功"不直接判漏洞成立；响应"失败"不直接判无问题。
