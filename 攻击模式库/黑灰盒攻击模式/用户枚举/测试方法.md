### 用户枚举攻击模式测试策略

> 🔒 **成立判定门槛**：登录 / 注册 / 找回密码接口对"已知存在用户"与"伪造不存在用户"返回**可区分**的响应——错误码、消息体、HTTP 状态、响应时长任一显著差异即自证。通用判定 / 置信度见 `通用黑灰盒测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. 探测请求构造（成对发，两次即可）

准备一对样本：

- **样本 A**：一个**已知存在**的用户名（已注册账号）。
- **样本 B**：一个**伪造不存在**的随机用户名（如 `nonexistent_<时间戳>`）。

针对登录 / 注册 / 找回密码三个接口，对每个接口分别用 A、B 各发一次（共最多 6 次），**密码或验证码字段都填错**（等量错误才能比响应差异）。

> 不要堆字典；只要能抓到差异，一对就够了。

发送后**记录每次响应头 `X-Request-Id`** + 响应时长。

#### 2. 响应观察（线索 + 直接证据）

成对对比 A 和 B 的响应，看以下维度是否可区分：

| 差异维度 | 示例 | 结论 |
|---|---|---|
| HTTP 状态 | A→200, B→404 | "确认" |
| 错误码 / 消息体 | A→「密码错误」, B→「用户不存在」 | "确认" |
| 响应时长 | A→1.2s, B→0.05s（>5× 差异稳定复现 2 次） | "确认" |
| 响应字段结构 | A 有额外字段（`failCount`、`lockUntil`），B 没有 | "确认" |
| 三维度都不可区分 | 文案统一、状态码一致、时长接近 | "无问题" |

时延差异要复现 ≥2 次，避免单次抖动误判。

#### 3. 后端核验 / 升级

- **响应自证已足够**：成对差异即判定证据。
- **http_req_query 反查**：用 `X-Request-Id` 看后端处理日志，区分 A/B 走的代码路径（A 进了"密码校验"、B 在"查用户"就 return）—— 解释差异来源。
- **结论不明**（响应文案统一但时延微差）→ 置信度"中" → 转白盒读登录 / 注册 / 找回密码处理代码（关键词 `findByUsername` / `bcrypt.compare` / `MessageDigest`），看是否对不存在用户也走了等长 hash 校验。

#### 4. 红线

- ❌ 不字典枚举（`top1000.txt` 之类）；一对样本足以判通道。
- ❌ 不要把已知存在的样本 A 用真实他人账号——用自己注册的测试账号或事先准备的沙箱账号。
- ❌ 时延差异必须稳定复现，不能拿单次结果下结论。
