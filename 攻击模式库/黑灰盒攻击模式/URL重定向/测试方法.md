### URL重定向攻击模式测试策略

> 🔒 **成立判定门槛**：必须有"服务端按用户输入跳转到外部域"的直接证据——响应 `Location` 头原样转发了攻击者控制的 URL，或浏览器最终落到外部域，即自证。通用判定 / 置信度见 `通用测试原则.md` 与 `核心策略/置信度与升级策略.md`。

#### 1. Payload 构造（一次一个，不堆）

获取一次正常带跳转的请求（登录后跳、外链中转、短链、回调），**仅替换跳转参数**（`redirect` / `url` / `target` / `next` / `returnUrl` 等）：

- 替换为外部域：`https://example.com`（中性、无副作用）。

只发一个 Payload，不要试 `//evil.com`、`/\\evil.com`、URL 编码绕过等变体——结论不明转白盒读跳转校验。

发送后**立即记录响应头 `X-Request-Id`**。

#### 2. 响应观察（线索 + 直接证据）

| 响应表现 | 可能含义 | 结论 |
|---|---|---|
| 响应 30x 且 `Location: https://example.com...` | URL 被原样转发（自证） | "确认" |
| 响应 200 + 页面 JS 跳转 `window.location = ...`（可控值） | 客户端跳转 | "确认" |
| 响应 30x 且 `Location` 指向本站路径或被改写 | 服务端做了白名单校验 | 记"防御观察" |
| 响应 4xx「invalid redirect」 | 前置校验命中 | "无问题" |

跟随 `Location` 多走一跳确认浏览器最终落点；如服务端只在第一跳做了校验、后续允许任意跳转，仍算成立。

#### 3. 后端核验 / 升级

- **响应自证已足够**：`Location` 头或 JS 跳转值就是判定证据。
- **myproxy-query 反查**：用 `X-Request-Id` 拿后端处理日志，看跳转目标的拼装位置。
- **结论不明** → 置信度"中" → 转白盒读跳转校验代码（关键词 `redirect` / `sendRedirect` / `Location` / `URL.parse` / `referer`），确认白名单 / 同站校验是否完备，再回灌一次。

#### 4. 红线

- ❌ 不堆绕过变体（`//evil.com`、`/\\evil.com`、URL 编码、Unicode 同形字符、子域同形）——一种命中即可下结论或转白盒。
- ❌ 不要把 Payload 域换成钓鱼站；用 `example.com` 这种中性占位。
- ❌ 响应"成功"不直接判漏洞成立；响应"失败"不直接判无问题。
