# 黑灰盒踩坑提醒

> 实操红线，与 `通用黑灰盒测试原则.md` 互补——后者讲方法，本文件讲红线。

## 一、Payload 必须打到目标服务端

黑灰盒攻击的载体永远是**发往被测系统**的请求（HTTP 字段值 / multipart / header / 上传文件内部条目等），由远端解析时触发。

- ✅ 把 Payload 放进发向目标接口的请求并发出。
- ❌ 在本地 shell 摆弄 Payload 字符（`ls ../`、`cat ../etc/passwd`），或只在本地构造而不发送——既不是攻击，也不产生证据。

判断口径：每个 Payload 都要能在 `http_req_query` 抓到它进入了发向远端的请求字段。各攻击模式的 `测试方法.md` 只补充该攻击专属的字段位置 / 编码细节，不重复本条。

## 二、后端搜索由小到大，不要从根开始

后端取证（`myssh find` / 日志检索 / 文件追踪）默认从**最小可能范围**开始；全盘搜（`find /` / `locate`）是兜底，慢、吵、还可能踩到敏感目录。

收敛优先级：响应或错误栈已暴露的路径片段 → `环境配置/测试环境.md` 写过的部署 / 上传 / 日志目录 → 从请求反查进程并看它的工作目录与已打开文件 → 按应用部署惯例推断 → 日志检索通常先于文件系统遍历。

判断口径：动手前能说清"搜哪个目录、为什么是这个"——说不清就再缩一轮。仍要全盘搜时，在报告"附带观察"里说明为何缩不下来。
