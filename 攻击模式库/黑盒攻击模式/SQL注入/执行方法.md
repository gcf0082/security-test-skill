### SQL 注入测试策略

#### 1. 识别条件
- 请求参数含**排序字段**（`orderBy`、`sort`、`sortField`）、**分页参数**（`limit`、`offset`）——这类参数通常拼接进 `ORDER BY` / `LIMIT`，预编译也无法保护。
- 搜索、筛选、ID 查询类接口。
- 响应中出现 `SQLException`、`You have an error in your SQL syntax`、`ORA-`、`PG::`、`sqlite3.`、`JDBC` 等数据库错误信息。

#### 2. 测试方法

**阶段一：布尔探测**
- 对数值型参数：`id=1 AND 1=1` vs `id=1 AND 1=2`
- 对字符串型参数：`name=foo' AND '1'='1` vs `name=foo' AND '1'='2`
- 响应差异（长度、内容、状态码）即疑似注入。

**阶段二：报错探测**
- 追加单引号 `'`、双引号 `"`、反斜杠 `\`，观察响应是否返回数据库错误堆栈。

**阶段三：时间盲注**
- MySQL：`id=1 AND SLEEP(5)`、`id=1' AND SLEEP(5)-- -`
- PostgreSQL：`id=1; SELECT pg_sleep(5)--`
- SQL Server：`id=1; WAITFOR DELAY '0:0:5'--`
- 对比响应时间是否出现 5 秒延迟。

**阶段四：ORDER BY 注入（高价值）**
- 许多项目对业务参数用预编译，但对排序字段直接拼接：
  - `sort=id` 正常 → `sort=id,(SELECT SLEEP(5))` 触发延迟
  - `sort=(CASE WHEN (1=1) THEN id ELSE name END)` 观察结果差异

**阶段五：分页注入**
- `limit=10` 正常 → `limit=10,1 UNION SELECT 1,2,...` 视列数拼接。

#### 3. 构造原则
- 先用**无副作用**的 Payload（`SELECT`、`SLEEP`）确认注入点，**禁止**直接 `DROP`、`UPDATE`、`DELETE`。
- 每次只改一个参数；先判定数据类型（数值/字符串）再选对应闭合方式。
- 时间盲注 sleep 值建议 5 秒，避开网络抖动与业务正常延迟。

#### 4. 判定依据
- 布尔 Payload 两种写法响应差异稳定。
- 时间 Payload 响应时间出现对应延迟（重复 2 次确认，排除网络波动）。
- 响应直接回显数据库版本、表名等（通过 `UNION SELECT` 或报错函数）。

#### 5. 常见误报与排除
- WAF 返回统一 500/403 时，两种 Payload 看起来都"异常"，需对照正常请求响应区分。
- 业务接口本身较慢（> 2s）时，需提高延迟阈值。
- `UNION SELECT` 列数不匹配会报错但并不代表没有注入，可切换报错/延时策略。
