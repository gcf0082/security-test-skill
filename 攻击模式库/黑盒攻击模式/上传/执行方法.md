### 上传测试策略

#### 1. 识别条件
- 表单包含文件选择控件（`<input type="file">`）。
- 接口请求 `Content-Type: multipart/form-data` 或含 `filename=` 字段。
- 业务点：头像、附件、导入、图片、证件照、富文本图片、备份恢复、模板上传。

#### 2. 测试方法

**阶段一：文件名路径穿越**
- **前置判断**：观察正常请求中文件名是否已被 URL 编码（出现 `%2e%2e%2f` 等），以此决定 Payload 构造方式。
- **Payload**：
  - 未编码：`../file.txt`（优先单层 `../`）
  - URL 编码：`%2e%2e%2ffile.txt`
  - 双重编码：`%252e%252e%252ffile.txt`
  - Windows：`..\file.txt`、`%2e%2e%5cfile.txt`
- **原则**：仅修改文件名字段，其他部分不变；尝试 1~2 次/Payload。
- **判定**：响应返回文件保存路径，或通过读回接口验证是否落到上级目录。

**阶段二：Zip Slip（压缩包解压穿越）**
当业务支持上传 ZIP、TAR 等压缩包并自动解压时：
- 使用工具构造包含 `../evil.txt`、`../../../etc/cron.d/attack` 等条目的 ZIP。
- 代码示例（Python）：
  ```python
  import zipfile
  with zipfile.ZipFile('evil.zip', 'w') as z:
      z.writestr('../evil.txt', 'pwned')
  ```
- **判定**：解压完成后，文件出现在预期目录之外。

**阶段三：扩展名绕过（上传可执行文件）**
按以下顺序尝试（强弱绕过）：
| 方法 | 示例 |
|------|------|
| 白名单扩展名配可执行内容 | 将 webshell 改名为 `shell.jpg`，利用业务包含漏洞触发解析 |
| 大小写混用 | `shell.PhP`、`shell.JsP` |
| 备用后缀 | `.phtml`、`.php3`、`.php5`、`.phar`、`.jspx`、`.asa`、`.cer` |
| 双扩展名 | `shell.php.jpg`、`shell.jpg.php` |
| NULL 截断 | `shell.php%00.jpg`（老旧 PHP/Java） |
| `.htaccess` / `web.config` | 改变所在目录的解析规则 |
| 畸形 multipart | 删除或伪造 `filename` 字段、`Content-Type` 字段 |

**阶段四：Content-Type 绕过**
- 上传 `shell.php` 时将 `Content-Type` 改为 `image/jpeg`。
- 或上传合法图片，追加 PHP 代码到文件尾（"图片马"），再结合本地文件包含。

**阶段五：文件大小与数量 DoS 自检**
- 不做实际压测，但应记录接口是否限制文件大小、是否限制上传频次。

#### 3. 构造原则
- **保留原始请求头**（Cookie、Authorization 等）。
- 每次只改一个维度（文件名 / 扩展名 / Content-Type / 内容）。
- **Webshell 必须使用无害特征**：仅返回固定字符串（如 `<?php echo 'TEST-' . md5('x'); ?>`），**禁止**上传真实 webshell。
- 上传完成后**主动清理**：如果业务提供删除接口则调用；否则在报告中记录以便人工清理。

#### 4. 判定依据
- 穿越后的文件可从超出范围的路径读取。
- 上传的 `.php`/`.jsp` 经 URL 访问后被解析执行（响应为执行结果而非源码）。
- 解压 ZIP 后目录外出现文件。
- 响应直接返回可预测/可访问的文件 URL。

#### 5. 常见误报与排除
- 上传成功 200 不等于攻击成功：要访问上传后的 URL 并验证内容是否按预期解析。
- 有些业务会对图片重新压缩（生成缩略图），导致"图片马"中的 PHP 代码被去除——此时需换渗透策略。
- CDN 前置可能缓存响应，测试时注意清缓存或加唯一查询参数。
