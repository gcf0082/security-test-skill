### 敏感信息白盒审计方法

> 适用场景：任务文字提到「敏感信息」「凭据」「密钥」「数据泄露」等关键词；
> 或黑灰盒已命中 `攻击模式特征/HTTP请求特征.md`「敏感信息泄露」、`攻击模式特征/Web页面特征.md`「敏感信息」条目，需要白盒补审定位源码落点。

> 🔒 判定通则（与 `核心策略/置信度与升级策略.md` 一致）：
>   - 看到证据 = 泄露成立；没看到 ≠ 不存在。
>   - 每条结论必须给出**文件路径 + 行号 + ≤10 行源码片段**作为铁证；只描述"可能存在"不算。
>   - 测试代码 / 文档样例 / 注释掉的旧代码 / 占位符（`changeme`、`xxxxxx`、`${ENV_VAR}`）→ 不算。

#### 1. 硬编码凭据 / 密钥

**搜索目标**
- 关键词：`password`、`passwd`、`secret`、`token`、`apiKey`、`accessKey`、`secretKey`、`jwt`、`salt`、`AES_KEY`、`RSA`、`BEGIN PRIVATE KEY`、`BEGIN RSA`、`BEGIN OPENSSH`
- 配置类文件后缀：`*.yml` / `*.yaml` / `*.properties` / `*.env` / `*.ini` / `*.json` / `*.toml`
- 写死的内网 / 外部带认证 URL：`localhost` 之外的硬编码 host + 用户名/密码段、`http(s)://user:pass@`

**判定与排除**
- 算：业务代码常量、被部署配置加载的值。
- 不算：`test/` / `*Test.java` / `*.spec.*` / mock 数据；占位符与环境变量引用。

#### 2. 响应 / DTO 字段泄露

**审计路径**：接口入口（Controller / Handler / Router）→ Service 返回类型 → 实体类 / DTO 字段。

**关注函数与注解**
- Java：`@RestController`、`@ResponseBody`、`@JsonProperty`、`@JsonIgnore`、`ObjectMapper`、`GsonBuilder`。
- Go：`json.Marshal` / struct `json:"..."` tag、`json:"-"` 排除标记。
- Python：`jsonify`、`.dict()` / `model_dump()`、Pydantic `Field(exclude=True)`。
- Node：`res.json(...)`、序列化时是否 `omit` / `pick`。

**检查项**
- 接口是否直接返回 `User` / `Account` 实体且含 `password` / `salt` / `loginToken` 等字段；
- 是否用 `@JsonIgnore` / `json:"-"` / `exclude` 等机制显式排除敏感字段；
- 手机号 / 身份证 / 银行卡 / 邮箱字段返回前是否经过脱敏函数（`mask*`、`desensitize*`）。

**判定**：返回字段含敏感字段且没有显式排除/脱敏 → 泄露成立。

#### 3. 日志 / 异常打印泄露

**搜索目标**
- 日志调用：`log.info` / `debug` / `warn` / `error`、`logger.*`、`System.out.print*`、`fmt.Print*`、`console.log`、`print`。
- 异常：`e.printStackTrace()`、`throw new RuntimeException(req.toString())`、`return e.getMessage()` 直接回写客户端。
- 请求体打印：`request.getParameterMap`、`JSON.stringify(req)`、`req.body` 整体 dump。

**检查项**
- 日志 / 异常参数中是否携带 `user.password`、`request.body`、`Authorization` 头、`token`、`idCard`、`bankCard`。
- 异常 message 是否被序列化回响应（向终端用户暴露 SQL 语句、文件路径、堆栈）。

**判定**：日志 / 异常路径上能搜到敏感字段 → 泄露成立；置信度按是否真实可达落盘的日志通道 / 客户端响应来定。

#### 4. 配置 / 版本控制泄露

**检查项**
- 项目根：`.env*`、`.git/`、`.svn/`、`*.bak`、`*.swp`、`dump.sql`、`backup.zip`、`*.pem`、`*.key`、`id_rsa*`。
- 配置文件：`application*.properties` / `application*.yml`、`web.xml`、`nginx.conf`、`docker-compose*.yml` 是否含明文密码 / 数据库连接串。
- 调试端点：`actuator/*`、`/debug`、`swagger-ui` 在生产 profile 下是否被关闭。
- 版本控制残留：`git log -p` 是否曾提交过含凭据的历史 commit（仅作为附带提示，不构成单独漏洞认定，避免误伤）。

**判定**：文件存在 + 含真实凭据 / 真实生产配置 → 成立；占位符或仅模板不算。

#### 输出

按 `攻击模式库/白盒攻击模式/通用白盒测试原则.md` "输出要求"段落写入 `报告/`，每条至少包含：
- 子类编号（1–4）
- 文件路径 + 行号 + ≤10 行源码片段
- 触发条件（该常量/字段在何种调用 / 接口 / 日志通道下被暴露）
- 黑白盒结合时附"下一步黑灰盒决策"——例如用泄露的 token 去打受保护接口、用泄露的 AK/SK 调内网服务等。
