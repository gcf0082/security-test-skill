# 白盒攻击模式说明

白盒攻击模式指在**具备被测项目源码**的前提下，通过代码审计方式定位漏洞并验证可利用性。

## 加载前提

- 当本次模式为 `白盒` 或 `黑白盒结合` 时加载本目录。
- 必须有可用的源码路径——优先级：任务描述显式路径 > `环境配置/测试环境.md` 覆盖值 > 项目内 `项目代码/`（默认）。逐级回落到的最终路径必须存在并非空。

前提不满足时本模式被**跳过**——不是自动转向黑灰盒。模式选择由任务决定，不由加载结果决定；若模式要求白盒但源码不可用，应中止并向用户反馈，而非偷偷降级。

## 通用流程

**汇点驱动 + 反向溯源**：先沿数据流传播 + 调用链找出入口函数能触达的全部**高危操作**（汇点），再反向看每个汇点的输入是否来自外部用户、外部输入是否经过有效校验。**所有高危操作无论是否构成风险都登记到报告**——目的是体现审计覆盖度，不是只记问题。

1. **入口定位**：根据任务给出的 URL / 接口或函数，从路由配置或框架注解反查对应的 Controller / Handler / 入口函数。

2. **高危操作枚举**：沿调用链与数据流从入口往下走，把所有触达的高危操作逐个记下来。典型类别（凡命中即视为高危，登记不漏）：
   - **文件操作**：`new File(path)` / `Files.write` / `open(path)` / `FileInputStream` / `move_uploaded_file` / 解压、压缩、删除、目录遍历
   - **命令 / 代码执行**：`Runtime.exec` / `ProcessBuilder` / `os.system` / `subprocess` / `eval` / `Function()` / 反序列化（`readObject` / `unserialize`）
   - **网络 / 外发调用**：`HttpClient` / `requests.*` / `URLConnection` / `fetch` / DNS 解析、SMTP / FTP / Webhook 发起
   - **SQL / NoSQL / LDAP / XPath / OS 注入面**：`Statement.execute` / 字符串拼接 query / `ldap.search` / `XPath.evaluate`
   - **模板 / HTML 输出**：模板引擎调用 / `innerHTML` / 未转义的响应字段
   - **认证 / 授权决策**：鉴权点本身（缺失或可绕过也是风险）
   - **加密 / 凭据处理**：密钥读取、`MessageDigest` / 弱算法、明文存储

3. **外部输入溯源**：对每个高危操作的关键入参，反向追到它的来源——
   - **直接来自 HTTP 请求**（`@RequestParam` / `@RequestBody` / `request.getParameter` / FormData / Header / URL 路径变量等） → 标"**疑似问题**"，进入 §4。
   - **来自常量 / 配置 / 内部强类型枚举 / 内部 ID** → 标"无外部输入"，登记为**无风险**。
   - **来自数据库但数据库内容来源于用户**（二阶注入面） → 同样标"疑似问题"。

4. **校验充分性确认**（只对疑似问题做）：核对输入到达汇点前是否经过**有效**校验——
   - 白名单 / 强类型 / 强格式（UUID、邮箱正则、路径白名单）/ 框架参数化（`PreparedStatement` / ORM 参数绑定）→ 充分，**降为"已审计、无风险"**。
   - 黑名单 / 关键词过滤 / 长度截断 → 多半不充分，标 **"中"** 或 **"高"**（视绕过难度）。
   - 完全无校验 / 仅前端校验 / 校验后又被改写 → 标 **"高"**。
   - 编码 / 转义（HTML escape / URL encode）→ 看是否在正确上下文中应用，错位等同无校验。

5. **利用链梳理**（仅对仍标风险等级的项做）：把"入口字段 → 中间方法 → 高危汇点 → 触发条件"串成可读链路；当本模式作为 `黑白盒结合` 补充时，附"下一步黑灰盒测试决策"。

## 专项审计（按需加载）

| 触发 | 子目录 |
|------|--------|
| 任务提到敏感信息 / 凭据 / 密钥泄露，或黑灰盒命中 `攻击模式特征/HTTP请求特征.md`「敏感信息泄露」/ `攻击模式特征/Web页面特征.md`「敏感信息」 | `敏感信息/测试方法.md` |

匹配后加载对应 `测试方法.md`，与第 1 节"通用流程"是**并列**关系——通用流程负责数据流类漏洞（注入 / 穿越 / SSRF / XSS），专项审计负责非数据流类（敏感信息等）。两者可同时启用，由实际任务决定。

## 输出要求

白盒分析结果写入 `报告/` 对应用例文件，含两层：

### A. 高危操作清单（必填，**不论是否有风险都要列**）

每条至少含：

| 字段 | 说明 |
|---|---|
| 汇点位置 | `<相对路径>:<行号>`（含函数名） |
| 操作类别 | 文件 / 命令 / 网络 / SQL / 模板 / 鉴权 / 加密 等 |
| 调用链 | 入口 → … → 该汇点的方法链 |
| 输入来源 | 外部 HTTP 字段名 / 常量 / 配置 / DB 二阶 |
| 校验情况 | 白名单 / 黑名单 / 框架参数化 / 无 等，含校验代码位置 |
| 风险等级 | **高 / 中 / 低 / 无风险**（无外部输入或校验充分时填"无风险"） |
| 备注 | 绕过条件、版本依赖、架构特性等 |

无外部输入 / 校验充分的汇点也要登记，标"无风险"——这是审计覆盖度证据。

### B. 风险项详细分析（仅对清单中风险等级 = 高 / 中 / 低 的项）

在 A 表基础上补：
- **关键源码片段**（文件路径 + 行号 + 上下文 5~20 行）
- **触发条件**（什么 Payload / 什么状态能触发）
- **与黑灰盒现象的对应关系**（已配合黑灰盒时填）
- **下一步黑灰盒测试决策**（仅 `黑白盒结合` 模式下必填）：据代码理解推导出来的新 Payload 方向或测试条件，交黑灰盒回灌验证。
